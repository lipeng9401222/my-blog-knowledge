# Core Web Vitals 实战：从指标理解到前端性能优化闭环

很多团队做性能优化时，最容易掉进两个极端：

- 只盯着 Lighthouse 分数，忽略真实用户体验
- 只说“页面有点慢”，但没有统一指标和治理方法

`Core Web Vitals` 的价值就在这里：它让“性能”从主观感受变成可以被测量、被定位、被回归验证的工程问题。

## 先记住 3 个核心指标

| 指标 | 关注点 | 建议阈值 |
| --- | --- | --- |
| LCP | 主要内容加载速度 | 小于等于 2.5 秒 |
| INP | 用户交互响应速度 | 小于等于 200 毫秒 |
| CLS | 页面布局稳定性 | 小于等于 0.1 |

如果要给业务团队做解释，可以这样理解：

- `LCP` 回答的是“用户什么时候看到主要内容”
- `INP` 回答的是“用户操作后多久得到响应”
- `CLS` 回答的是“页面会不会乱跳”

## 为什么只看实验室数据不够

Lighthouse、DevTools Performance 这类工具非常重要，但它们更多是“受控环境下的体检报告”。

真实线上体验还会受到这些因素影响：

- 用户设备性能差异
- 网络状况波动
- 页面是否命中缓存
- 第三方脚本是否临时抖动
- 同一路由是否承载了不同内容规模

所以更稳妥的做法是把性能治理分成两层：

1. 实验室数据：用于本地分析、复现问题、验证优化效果
2. 真实用户监控：用于观察线上分布、识别高影响问题

## 一套更靠谱的性能优化闭环

### 第一步：先确认问题在哪个指标

不要一上来就做图片压缩、代码拆分、缓存优化一条龙。先看问题属于：

- 首屏主要内容慢
- 交互响应慢
- 页面布局跳动

不同指标背后的根因完全不同。

### 第二步：把问题定位到资源、线程或布局

通常可以继续往下拆成 3 类：

- 资源问题：图片、字体、脚本、接口响应
- 主线程问题：长任务、重计算、频繁重渲染
- 布局问题：图片尺寸不固定、异步插入内容、广告位占位不足

### 第三步：只做和指标强相关的优化

例如：

- `LCP` 差，不代表一定要先做虚拟列表
- `CLS` 差，不代表压缩 JS 包就能解决
- `INP` 差，也不一定是接口慢，可能是前端主线程被卡住了

### 第四步：优化后必须回归验证

至少验证 3 件事：

- 指标是否真的改善
- 是否引入了新的回归问题
- 是否在慢设备或弱网下也成立

### 第五步：把规则写进工程体系

如果每次都靠人肉盯 Lighthouse，性能治理很难长期持续。更推荐把它变成：

- 图片尺寸与格式规范
- 组件渲染边界约定
- 性能预算
- 发布前检查项

## LCP 优化：优先解决“首屏主内容晚出现”

`LCP` 最常见的问题不是“页面整体都很慢”，而是“最关键那块内容太晚呈现”。

### 常见根因

- 首屏大图过大，且没有合理压缩
- 首屏依赖的 CSS / JS 阻塞渲染
- 服务端响应慢，HTML 迟迟没回来
- 首屏数据请求串行，导致主内容等待多个接口
- 字体加载策略不当，文本延迟可见

### 常见优化手段

#### 1. 优先处理首屏图片

- 给首屏大图使用更合适的格式，例如 WebP / AVIF
- 根据展示尺寸输出资源，而不是直接塞原图
- 对首屏关键图片使用更积极的加载策略

#### 2. 降低渲染阻塞资源

- 拆分不属于首屏的脚本
- 控制首屏样式体积
- 避免把大量第三方脚本堆在入口页

#### 3. 减少串行等待

如果首屏必须等多个接口都返回，LCP 通常不会好看。要尽量把：

- 首屏必要数据
- 次屏补充数据
- 低优先级增强内容

分层处理，而不是一次全等完。

## INP 优化：优先解决“操作后没反应”

现在很多页面“看起来首屏还行”，但一点击就卡，根因大多在 `INP`。

### 常见根因

- 点击后触发大量同步计算
- 组件树更新范围过大
- 长列表渲染或大表格重绘
- 输入框联动逻辑过重
- 第三方 SDK 或埋点阻塞主线程

### 常见优化手段

#### 1. 拆长任务

如果某次点击后紧接着执行了大段同步逻辑，浏览器就没机会及时响应下一次输入。

思路通常是：

- 拆分大计算
- 延后低优先级任务
- 把不需要同步完成的逻辑异步化

#### 2. 缩小更新范围

前端项目里最常见的卡顿，不是某一行代码太慢，而是一次操作引发了整片页面重渲染。

可以重点排查：

- 状态是否提升过高
- 是否存在无意义的父组件刷新
- 列表项是否能局部更新

#### 3. 控制交互链路上的副作用

点击或输入事件上不要同时做这些事情：

- 立刻打多个埋点
- 同步格式化大量数据
- 计算复杂筛选条件
- 触发过深的 watch / effect 链条

## CLS 优化：优先解决“页面乱跳”

`CLS` 其实很好解释：用户已经开始阅读或准备点击时，页面突然挪动了位置。

### 常见根因

- 图片、视频、广告位没有提前占位
- 异步内容插入到已有内容上方
- 字体切换导致文本重排
- 动画直接改 `top` / `left` 触发布局

### 常见优化手段

#### 1. 给媒体资源预留稳定尺寸

这是最常见、也最容易见效的一条。

```html
<img
  src="/banner.webp"
  alt="活动横幅"
  width="1200"
  height="600"
/>
```

#### 2. 异步内容不要强行插队

如果推荐位、通知条、广告模块一定要异步加载，尽量提前预留空间，而不是加载后把正文整体往下推。

#### 3. 优先使用 transform 做动画

涉及位移时，优先考虑 `transform`，减少直接修改布局属性导致的重排。

## 指标和优化手段不要错配

下面这个表适合团队内部快速对齐。

| 问题表现 | 更可能受影响的指标 | 优先排查方向 |
| --- | --- | --- |
| 首屏大图很晚出来 | LCP | 图片、接口、阻塞资源 |
| 点击筛选后界面卡住 | INP | 长任务、状态更新范围 |
| 页面加载后按钮位置跳动 | CLS | 占位、异步插入、字体切换 |
| Lighthouse 高，线上用户还说慢 | 真实体验指标 | RUM、弱网、低端机表现 |

## 建议尽早接入真实用户监控

如果项目已经有一定访问量，建议尽快把 `web-vitals` 接到监控系统里。

```javascript
import { onCLS, onINP, onLCP } from 'web-vitals'

function reportMetric(metric) {
  navigator.sendBeacon(
    '/monitor/web-vitals',
    JSON.stringify({
      name: metric.name,
      value: metric.value,
      id: metric.id
    })
  )
}

onCLS(reportMetric)
onINP(reportMetric)
onLCP(reportMetric)
```

有了真实数据之后，优化顺序就可以更有依据，例如优先处理：

- 流量最高的页面
- 指标最差且影响转化的页面
- 某个设备类型或网络环境下明显异常的页面

## 做性能优化时最常见的误区

- 只看一次 Lighthouse 结果，就宣布优化完成
- 没分清是后端慢、资源慢还是前端主线程卡
- 优化点做了很多，却没有基线和回归数据
- 为了追求分数，把业务体验牺牲掉

性能优化不是比赛分数，而是持续改善用户完成任务的成本。

## 一份可执行的性能检查清单

上线前至少可以快速过一遍：

- 首屏主图是否按展示尺寸输出
- 是否存在明显的阻塞脚本和无关第三方资源
- 页面交互是否会触发长任务
- 大列表是否有虚拟化或分页策略
- 图片、广告位、异步模块是否有稳定占位
- 是否记录了真实用户指标

## 总结

`Core Web Vitals` 的真正价值，不只是告诉你“页面快不快”，而是帮助团队建立一套从发现问题、定位问题、修复问题到防止回归的统一语言。

如果只能记住一句话，我会建议记住这个顺序：

先看指标，再找根因；先做高相关优化，再用数据验证；最后把经验写进工程规范。
