# 05 项目实战路线

## 为什么必须做项目

截图岗位写了 2 年以上 AI/ML 开发经验。对于没有对应工作经验的人来说，项目作品是弥补经验差距的关键。

项目不能只是“跑通教程”，必须满足：

1. 有业务背景。
2. 有数据处理。
3. 有模型或大模型能力。
4. 有评估指标。
5. 有 API 或交付形态。
6. 有 README 和技术文档。
7. 能在面试中讲清楚取舍和失败案例。

## 项目总览

| 项目 | 核心能力 | 建议周期 |
|---|---|---:|
| 工业时序数据清洗管道 | pandas、NumPy、时序数据、特征工程 | 3 到 4 周 |
| 预测性维护模型 | scikit-learn、PyTorch、模型评估 | 4 到 6 周 |
| 工业规程 RAG 知识库 | 大模型 API、LangChain/LlamaIndex、向量库 | 4 到 6 周 |
| 模型服务与集成 Demo | FastAPI、Docker、REST API、工业系统模拟 | 4 到 5 周 |

## 项目交付补充标准

所有项目都按同一套最低标准沉淀，避免只做成“跑过一次的练习”。

### 最小验收命令

每个项目 README 至少提供 3 类命令：

~~~bash
python --version
pip install -r requirements.txt
python src/main.py --help
~~~

如果是 API 项目，额外提供：

~~~bash
uvicorn app.main:app --reload
curl http://127.0.0.1:8000/health
~~~

如果是 Docker 项目，额外提供：

~~~bash
docker build -t industrial-ai-demo:0.1 .
docker run --rm -p 8000:8000 industrial-ai-demo:0.1
~~~

### README 必备结构

~~~markdown
# 项目名称

## 业务背景

## 数据来源和字段说明

## 运行方式

## 核心流程

## 结果和指标

## 失败案例或改进方向
~~~

### 可展示交付物

| 类型 | 最低要求 |
|---|---|
| 代码 | 核心逻辑模块化，不只放 Notebook |
| 数据说明 | 字段、样本量、缺失率、标签定义清楚 |
| 报告 | 至少包含指标、图表、业务解释 |
| 服务 | API 有健康检查、输入输出示例、错误处理 |
| 简历材料 | 每个项目准备 1 分钟讲解和 3 个追问回答 |

## 项目 1：工业时序数据清洗管道

### 项目目标

构建一个可复用的数据处理管道，处理类似 DCS/SCADA 的传感器时序数据，输出高质量建模数据集。

### 业务背景

工厂设备持续产生温度、压力、流量、振动、电流等数据。原始数据可能存在缺失、重复、时间乱序、异常尖峰、单位不一致等问题。模型训练前必须先完成清洗和特征工程。

### 技术栈

| 技术 | 用途 |
|---|---|
| Python | 脚本开发 |
| pandas | 数据清洗和聚合 |
| NumPy | 数值计算 |
| Matplotlib/Plotly | 数据可视化 |
| Markdown | 数据质量报告 |

### 数据字段建议

| 字段 | 类型 | 说明 |
|---|---|---|
| timestamp | datetime | 采集时间 |
| equipment_id | string | 设备编号 |
| tag_name | string | 点位名称 |
| temperature | float | 温度 |
| pressure | float | 压力 |
| flow | float | 流量 |
| vibration | float | 振动 |
| current | float | 电流 |
| status | string | 设备状态 |

### 功能清单

| 功能 | 要求 |
|---|---|
| 数据读取 | 支持 CSV，Excel 可选 |
| 时间处理 | 时间戳解析、排序、去重 |
| 缺失处理 | 统计缺失率，按策略填充或删除 |
| 异常标记 | 阈值、3-sigma、箱线图规则 |
| 重采样 | 支持 1 分钟、5 分钟、1 小时 |
| 特征工程 | 滑窗均值、最大值、最小值、标准差、变化率 |
| 数据输出 | 输出清洗数据、特征数据、质量报告 |

```text
industrial-data-cleaner/
  README.md
  data/
    raw/
    processed/
  notebooks/
    01_eda.ipynb
  src/
    generate_data.py
    load_data.py
    clean_data.py
    build_features.py
    report.py
  reports/
    data_quality_report.md
  requirements.txt
```

### 验收标准

| 验收项 | 标准 |
|---|---|
| 可运行 | 一条命令能从原始数据生成特征数据 |
| 可解释 | 报告说明处理了多少缺失、异常、重复 |
| 可复用 | 换一份同结构数据仍能运行 |
| 可展示 | 有 EDA 图表和数据质量结论 |

### 简历描述示例

> 构建工业传感器时序数据清洗管道，支持时间戳对齐、缺失值处理、异常点标记、重采样和滑窗特征生成，输出数据质量报告和模型训练数据集，为后续预测性维护模型提供标准化数据输入。

### 面试讲解重点

1. 工业时序数据有哪些质量问题。
2. 为什么不能随便删除异常值。
3. 滑窗特征如何构造。
4. 如何避免标签泄漏。
5. 数据质量报告包含什么。

## 项目 2：预测性维护模型

### 项目目标

基于设备历史传感器数据，预测未来一段时间内的故障风险，并输出风险等级和影响因素。

### 业务背景

设备故障会造成停机、质量波动和维修成本。预测性维护希望在故障发生前提前预警，让运维人员安排检查和维修。

### 技术栈

| 技术 | 用途 |
|---|---|
| pandas/NumPy | 数据处理 |
| scikit-learn | 传统 ML baseline |
| PyTorch | LSTM/Transformer 入门模型 |
| Matplotlib/Plotly | 结果可视化 |
| joblib | 模型保存 |

### 建模任务

建议优先做分类任务：

| 输入 | 输出 |
|---|---|
| 过去 30 分钟传感器特征 | 未来 10 分钟是否有故障风险 |

可扩展为回归任务：

| 输入 | 输出 |
|---|---|
| 过去 N 分钟数据 | 未来温度、振动、压力或质量指标 |

### 模型方案

| 模型 | 作用 |
|---|---|
| 规则阈值 | 最简单业务 baseline |
| Logistic Regression | 可解释 baseline |
| Random Forest | 强 baseline |
| Gradient Boosting | 性能 baseline |
| LSTM | 深度时序模型 |
| Transformer Encoder | 加分对比 |

### 指标

| 任务 | 指标 |
|---|---|
| 分类 | Precision、Recall、F1、ROC-AUC、混淆矩阵 |
| 回归 | MAE、RMSE、R2 |
| 工业业务 | 漏报率、误报率、提前预警时间 |

### 推荐目录

```text
predictive-maintenance/
  README.md
  data/
  notebooks/
    01_feature_analysis.ipynb
    02_model_compare.ipynb
  src/
    build_dataset.py
    train_baseline.py
    train_lstm.py
    evaluate.py
    predict.py
  models/
  reports/
    model_report.md
  requirements.txt
```

### 验收标准

| 验收项 | 标准 |
|---|---|
| 数据集 | 标签定义清楚，时间切分合理 |
| 模型 | 至少 3 个传统 ML 模型和 1 个深度模型 |
| 指标 | 不只看准确率，必须有召回和混淆矩阵 |
| 解释 | 能说明误报和漏报的业务影响 |
| 输出 | 保存最佳模型，支持单条数据预测 |

### 简历描述示例

> 基于工业设备时序数据构建预测性维护模型，使用滑窗统计、滞后特征和变化率特征表征设备状态，对比 Logistic Regression、Random Forest、Gradient Boosting 与 LSTM 模型，并从召回率、误报率和提前预警时间角度评估模型效果。

### 面试讲解重点

1. 标签如何定义。
2. 为什么按时间切分训练集和测试集。
3. 类别不平衡怎么处理。
4. 为什么工业故障预测更关注漏报。
5. 深度模型是否一定优于传统模型。

## 项目 3：工业规程 RAG 知识库

### 项目目标

构建一个基于工业规程、设备手册、维护案例的私域知识库，支持自然语言问答，并给出引用来源。

### 业务背景

工艺、运维、安全规程通常分散在 Word、PDF、Excel 和内部文档中。新人和一线人员很难快速找到答案。RAG 可以把文档检索和大模型生成结合，提升知识查询效率。

### 技术栈

| 技术 | 用途 |
|---|---|
| Python | 主开发语言 |
| LangChain 或 LlamaIndex | RAG 编排 |
| DeepSeek 或 Qwen | 大模型问答 |
| Embedding 模型 | 文本向量化 |
| Chroma 或 FAISS | 向量检索 |
| FastAPI | 可选，封装问答接口 |

### 数据来源

可使用：

1. 自己编写的模拟设备维护手册。
2. 公开设备说明文档。
3. 安全操作规程样例。
4. 项目中的需求文档或技术文档。

注意：不要上传包含敏感信息的企业内部文档到外部 API。

### 核心流程

```text
文档收集 -> 文档清洗 -> 文档切分 -> 向量化 -> 向量库入库 -> 检索 -> Prompt 组装 -> 大模型回答 -> 返回引用
```

### 功能清单

| 功能 | 要求 |
|---|---|
| 文档加载 | 支持 Markdown 或 TXT，PDF/Word 可作为加分 |
| 文档切分 | 保留标题、段落和来源 |
| 向量入库 | 每个 chunk 有 metadata |
| 检索 | 返回 top_k 相关片段 |
| 问答 | 基于片段回答 |
| 引用 | 返回文件名和段落 |
| 兜底 | 检索不到时明确说没有依据 |
| 评测 | 准备 30 个问题 |

### 推荐目录

```text
industrial-rag-kb/
  README.md
  docs_source/
  data/
    chunks/
    vector_store/
  src/
    clean_documents.py
    build_index.py
    retrieve.py
    rag_query.py
    evaluate.py
  eval/
    questions.csv
    results.md
  requirements.txt
```

### 验收标准

| 验收项 | 标准 |
|---|---|
| 入库 | 至少 5 份文档 |
| 检索 | 查询能返回相关片段 |
| 问答 | 答案基于文档，不自由发挥 |
| 引用 | 每个答案有来源 |
| 评测 | 有问题集和失败案例 |

### 简历描述示例

> 构建工业规程 RAG 知识库，完成设备手册和操作规程的文档清洗、分块、向量化和检索问答，支持返回引用来源，并通过自建问题集评估检索命中率和回答正确率，降低大模型在规程问答场景中的幻觉风险。

### 面试讲解重点

1. 为什么 RAG 比直接问大模型更适合企业知识库。
2. 文档切分策略如何设计。
3. 如何保证答案可追溯。
4. 检索不到时怎么处理。
5. 如何评估 RAG 效果。

## 项目 4：模型服务与集成 Demo

### 项目目标

把预测模型和 RAG 知识库封装成 FastAPI 服务，并用 Docker 本地部署，模拟对接工业平台。

### 业务背景

模型只有被业务系统调用才有价值。工业平台通常需要通过 API 获取预测结果、报警等级、知识库答案，再展示给运维或工艺人员。

### 技术栈

| 技术 | 用途 |
|---|---|
| FastAPI | API 服务 |
| Pydantic | 请求体验证 |
| scikit-learn/joblib | 模型加载 |
| LangChain/LlamaIndex | RAG 调用 |
| Docker | 容器化 |
| Swagger | 接口文档 |

### 接口设计

#### `GET /health`

作用：服务健康检查。

响应示例：

```json
{
  "status": "ok",
  "service": "industrial-ai-service"
}
```

#### `GET /version`

作用：返回服务和模型版本。

响应示例：

```json
{
  "service_version": "0.1.0",
  "predict_model_version": "maintenance_rf_v1",
  "rag_index_version": "kb_2026_06"
}
```

#### `POST /predict`

作用：输入传感器特征，返回故障风险。

请求示例：

```json
{
  "equipment_id": "PUMP_001",
  "features": {
    "temperature_mean_30m": 82.4,
    "pressure_mean_30m": 1.25,
    "vibration_std_30m": 0.18,
    "current_diff_5m": 1.7
  }
}
```

响应示例：

```json
{
  "equipment_id": "PUMP_001",
  "risk_score": 0.82,
  "risk_level": "warning",
  "reason": ["temperature_mean_30m high", "vibration_std_30m abnormal"]
}
```

#### `POST /rag/query`

作用：输入工业规程问题，返回知识库答案。

请求示例：

```json
{
  "question": "PUMP_001 轴承温度持续升高时应该如何处理？",
  "top_k": 3
}
```

响应示例：

```json
{
  "answer": "应先检查润滑状态、轴承磨损和冷却条件，并根据规程执行降载或停机检查。",
  "sources": [
    {
      "document": "pump_maintenance_manual.md",
      "chunk_id": "pump_maintenance_manual:12"
    }
  ]
}
```

### 验收标准

| 验收项 | 标准 |
|---|---|
| 接口 | `/health`、`/version`、`/predict`、`/rag/query` 可用 |
| 文档 | Swagger 和 README 都有调用说明 |
| 模型 | `/predict` 调用真实模型或可解释 mock |
| RAG | `/rag/query` 返回答案和来源 |
| Docker | 能构建镜像并运行服务 |
| 稳定性 | 有异常处理和日志 |

### 简历描述示例

> 使用 FastAPI 将预测性维护模型和工业规程 RAG 知识库封装为 RESTful API，提供健康检查、模型预测、知识库问答等接口，并通过 Docker 完成本地容器化部署，模拟 MES/工业互联网平台调用模型服务的集成流程。

### 面试讲解重点

1. 为什么要把模型封装成 API。
2. 请求体和响应体如何设计。
3. 模型如何加载，是否每次请求都加载。
4. 如何处理错误、日志、超时。
5. Docker 解决了什么问题。

## 综合作品集包装

最终建议把 4 个项目串成一个故事：

```text
工业现场产生传感器数据和规程文档
  -> 清洗时序数据并构造特征
  -> 训练预测性维护模型
  -> 构建规程 RAG 知识库
  -> 用 FastAPI 封装模型和知识库
  -> Docker 部署
  -> 模拟 MES/工业平台调用
```

作品集总标题可写：

> 工业设备预测性维护与智能知识库一体化 Demo

## 项目评分表

每个项目完成后按 100 分自评：

| 维度 | 分值 | 标准 |
|---|---:|---|
| 业务背景 | 10 | 能说明为什么做 |
| 数据处理 | 20 | 数据清洗和特征工程完整 |
| 模型/AI 能力 | 20 | 有模型、RAG 或 API 调用 |
| 评估 | 15 | 有指标、对比、失败案例 |
| 工程化 | 15 | 模块化、日志、配置、可运行 |
| 文档 | 10 | README、接口说明、报告 |
| 面试表达 | 10 | 能讲清技术取舍和业务价值 |

80 分以上才建议写进简历。
