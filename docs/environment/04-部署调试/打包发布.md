## 前端打包

### 组件化模式打包发布和更新

#### 检查配置项是否正确(如已确认，请跳过此步骤)
- 适用于vue组件化工程，请先确保[[vue组件化工程环境](https://fdoc.epoint.com.cn/onlinedoc/rest/d/A3YRVf "vue组件化工程环境")](?file=012-快速入门/001-安装环境#vue组件化环境安装)


#### 编译打包
- 如果仅需要vue组件化工程发布，直接查看下一节内容即可，发布命令会启动进行一次编译。
- 执行以下命令将对工作空间内所有的vue组件化工程和vue-web工程进行编译打包：<br>

```bash
# 先pnpm i 确保依赖包是最新的，为保证Install时相关组件是最新的，一般执行 pnpm up && pnpm install
# 如已执行，请跳过此步骤
pnpm up && pnpm install

# 通过eui-cli build一次将所有组件全部build
eui-cli build
```

#### 发布到仓库
- 关于发布命令的具体参数，请参考[vue组件化发包](http://192.168.219.170/docs/vue/latest/frame/guide/componentization-developer-guide.html#%E5%8F%91%E5%B8%83%E5%8C%85)
- <font style="color:red">一般只建议将组件化的包发布到仓库，不建议将vue-web发布到仓库，其定义和后端的epoint-web是一致的，仅是运行容器</font>。
- 发布包只能一个个在 <font style="color:red">vue组件化工程根目录下</font>执行发布命令，基于安全性考虑不支持批量发布。
- 首次发布推荐用`pnpm run release`执行(因为`eui-cli` 脚手架命令必定会迭代版本号，首次是不需要迭代的)：<br>

```bash
# 执行前确保已在该组件的根目录下，否则无效
pnpm run release

# 如果需要忽略git检测，添加--no-git-checks
pnpm run release --no-git-checks

# 如果首次发布的snapshot版本，建议加tag
# 推荐tag的命名方式：将原版本号进行倒装，比如10.0.0-snapshot.x系列的 tag名字为 snapshot-10.0.0
pnpm run release --tag snapshot-10.0.0
```

- 后续迭代版本号进行发布，推荐使用`eui-cli`脚手架，脚手架会自动修改`package.json`进行版本号更新，并自动添加git记录<br>

```bash
# 如果希望交互式来发布，执行时不要加附加参数即可，直接publish
eui-cli publish

# 以下为一些示例，具体请根据情况做判断

# 如果希望发布时忽略git检测，可以加--no-git-checks参数，有时候我们临时改了本地代码发布了给别人做验证，但不想提交git
eui-cli publish --no-git-checks

# 如果希望迭代发布最大的版本号，比如 开始是 9.5.0 需要将主版本号(major) 从9 升级到 10，执行以下，版本号将升级为10.0.0
eui-cli publish --increment major

# 如果希望迭代次版本号，比如 开始是 9.5.2 需要将次版本号(minor) 从5 升级到 6，执行以下，版本号将升级为9.6.0
eui-cli publish --increment minor

# 如果希望迭代小版本号，比如 开始是 9.5.2 需要将小版本号(patch) 从2 升级到 3，执行以下，版本号将升级为9.5.3
eui-cli publish --increment patch

# 如果希望升级预发布版本号，(snapshot,sp，beta等等都属于其中)
# 比如 9.5.6-snapshot，想要发布一个新的快照版本，执行以下，将升级为9.5.6-snapshot.0,后续执行会将最后一位递增
# 命令会自动 添加tag,通过tag将snapshot指向到最新，因为使用方在用预发布版本号的时候并不想随时改package.json
# 推荐tag的命名方式：将原版本号进行倒装，比如9.5.6-snapshot.x系列的 tag名字为 snapshot-9.5.6
# 这样 使用方package.json里对应组件的版本号直接设置为snapshot-9.5.6即可，通过eui-cli update 后会自动更新到最新的快照
eui-cli publish --increment prerelease
# 如果 希望自己指定tag，可以直接直接自己指定一个想要的名字，snap-9_5_6 ，如果自己指定，以指定的为准
eui-cli publish --increment prerelease --silent --tag snap-9_5_6
```

#### 更新包
- 关于更新命令使用，请参考[vue组件化更新](http://192.168.219.170/docs/vue/latest/frame/guide/componentization-developer-guide.html#%E6%9B%B4%E6%96%B0%E5%8C%85)
- 更新指的是将 vue-web工程的package.json引用的组件包更新，因此<font style="color:red">此命令必须在vue-web的根目录下执行</font>
- 由于pnpm有2级缓存，vue更新都推荐使用脚手架进行，脚手架会自动清缓存，避免未更新到位导致一些奇怪问题<br>

```bash
# 快速更新，仅能更新一些版本号指定为tag的组件，但胜在速度快，并且不强制在vue-web下执行，只要保证在工作空间内
pnpm up && pnpm install

# 如果希望交互式来更新
eui-cli update

# 如果不希望把package.json里版本号自动刷新到最新(大部分场景是这种情况)，仅更新包到最新
eui-cli update --no-save

# 如果希望把一写其他组件依赖版本也升级到最新且静默，则:
eui-cli update --silent

# 如果希望所有的都强制更新，需要加--all ，比如以下
eui-cli update --no-save --all

```

### 流水线模式打包发布

利用集成开发平台的流水线能力，可以做到流水线模式发包。

#### 添加代码仓库（如已存在，跳过此步骤）

登录集成开发平台，将你需要流水线发布的代码所在仓库注册到集成开发平台的代码仓库内：

![011-1755828377720.png](https://fdoc.epoint.com.cn/onlinedoc/rest/frame/base/attach/attachAction/getContent?isCommondto=true&attachGuid=029233b1-2827-4df0-b955-312589c51958)

如果没有凭据可供选择，需要手动在右上角自行配置，具体可参考[配置仓库凭证](https://fdoc.epoint.com.cn/onlinedoc/rest/d/6jEVVf)。

#### 添加流水线计划

在录入代码仓库后，前往构建计划，根据对应仓库下的分支添加构建计划：

![011-1755829202407.png](https://fdoc.epoint.com.cn/onlinedoc/rest/frame/base/attach/attachAction/getContent?isCommondto=true&attachGuid=8d172f6d-de7d-48b8-9585-b8d0082b1cc4)

然后修改内部的构建编排逻辑，仅保留拉取代码、构建工程步骤：

![011-1755830055206.png](https://fdoc.epoint.com.cn/onlinedoc/rest/frame/base/attach/attachAction/getContent?isCommondto=true&attachGuid=301af056-35f9-40b5-af6e-c69117a61673)

在拉取代码步骤内，如果你的仓库里前后端代码混在一块，仅需要拉取你需要发包的前端根目录，在checkOut方法传参的extensions参数下，填写`[$class: 'SparseCheckoutPaths', sparseCheckoutPaths: [[path: 'abc/']]]`其中`abc/`指代对应你指定的根目录。

在构建工程步骤内，通过调用集成开发平台统一封装的`npmScript`方法进行处理，传入2个参数：
- 第一个参数表示执行命令，对于发包而言我们一般使用`eui-cli publish`进行处理；考虑到集成开发平台的node版本，我们额外需要添加前缀`nvm exec 20.17.0 pnpm i && nvm exec 20.17.0 `进行指定；考虑到流水线一般不进行交互，需要指定参数` --no-git-checks --silent`；一般默认的命令即为`nvm exec 20.17.0 pnpm i && nvm exec 20.17.0 eui-cli publish --no-git-checks --silent`。如果有其他参数需求可自行追加处理。
- 第二个参数表示指定的根目录，一般而言和上面指定的根目录保持一致，如`abc/`。

下面是框架一个组件发布流水线构建编排示例可供参考：

```
@Library('epoint-tmp@dev') _

pipeline{
    agent any

    environment {
        // 脚本文件模版类型
        CLOUDDEV_TEMPLATE="CLOUDDEV_TEMPLATE_WAR"
        // 集成开发平台的版本
        CLOUDDEV_VERSION="2.0"
        // 流水线标识
        CLOUDDEV_PIPELINE_GUID="${CLOUDDEV_PIPELINE_GUID}"
        // 流水线所属项目标识
        CLOUDDEV_PROJECT_GUID="${CLOUDDEV_PROJECT_GUID}"
        // 流水线所属项目编号
        CLOUDDEV_PROJECT_NUM="${CLOUDDEV_PROJECT_NUM}"
        // 代码仓库路径（首次新增）: http://192.168.0.200/frame-public-group/epoint-web.git
        CLOUDDEV_CODE_URL="${CLOUDDEV_CODE_URL}"
        // Git分支名称（首次新增）: 952V
        CLOUDDEV_GIT_BRANCH="${CLOUDDEV_GIT_BRANCH}"
        // 代码仓库凭证（默认，代码仓库配的地址是什么类型就是什么类型）
        CLOUDDEV_CREDENTIALS_ID="${CLOUDDEV_CREDENTIALS_ID}"
        // 代码仓库凭证（GIT仓库的凭据）
        CLOUDDEV_CREDENTIALS_ID_GIT="${CLOUDDEV_CREDENTIALS_ID_GIT}"
        // 代码仓库凭证（SVN仓库的凭据）
        CLOUDDEV_CREDENTIALS_ID_SVN="${CLOUDDEV_CREDENTIALS_ID_SVN}"
        // Maven凭证
        CLOUDDEV_MAVEN_CREDENTIALS_ID="${CLOUDDEV_MAVEN_CREDENTIALS_ID}"
    }

    options {
        // 不允许并行执行此任务;即同一时间内容只允许存在一次正在执行的构建
        disableConcurrentBuilds()
        // 日志输出带有时间戳
        timestamps()
        // 设置流水线执行的超时时间为30分钟
        timeout(60)
    }

    stages{
        stage("环境配置") {
            steps {
                script {
                    // 启用debug模式, 功能尚未稳定, 请不要关闭
                    config.enableDebugModel()
                    // 配置部署包的更新模式OFF, FULL
                    // config.setUpdateMode('OFF')
                    // 配置在启动更新模式的情况下是否产出全量部署包
                    // config.setOutputWar(false)
                    // 不建议每次都开启, 配置每次构建之前自动清理仓库, 使用第二个参数指定仓库内的具体路径
                    // config.setAutoCleanMavenRepo(true, '')
                    // 配置在本次构建过程中使用的Maven凭证, 用于发包
                    // config.setMavenCredential('oa登录帐号')
                }
            }
        }

        stage("拉取代码") {
            steps {
                cleanWs()
                checkout([$class: 'GitSCM', branches: [[name: CLOUDDEV_GIT_BRANCH]], extensions: [[$class: 'SparseCheckoutPaths', sparseCheckoutPaths: [[path: 'epoint-minivue/']]]], userRemoteConfigs: [[credentialsId: CLOUDDEV_CREDENTIALS_ID, url: CLOUDDEV_CODE_URL]]])

            }
        }

        stage("构建工程") {
            steps {
               script{
                npmScript('nvm exec 20.17.0 pnpm i && nvm exec 20.17.0 eui-cli publish --no-git-checks --silent --increment=major','epoint-minivue')
               }
            }
        }
    }
}
```

## FAQ
### 发布账号如何进行绑定？

- vue技术架构下，前端开发的工程也是一个个组件，也可以发布到公司npm仓库中进行统一管理。（前提你的账号需要有权限,账号和后端maven是一样的）
- 在遇到无法发布到仓库提示无权限的时候，也可能是该发布地址本地没有登录绑定过发布源，按一下做一遍，然后切换回使用源就行了

```bash
# 新增发布源 ，如果是框架最后是frame-npm，如果是交易就是 dzjy-npm； 定义个名字比如epPublish
nrm add epPublish http://192.168.0.99:8081/nexus/repository/frame-npm/
# 切换到新点源
nrm use epPublish
# 登录，按提示输入账密
npm login

# 可以测试是否连上
nrm test epframe

# 切换回 原来的发布源， 即地址是npmpublic的那个名字，一般是epoint
nrm use epoint

# 接下来就可以正常发布 在你仓库的根目录下eui-cli publish 或者pnpm run release  具体上文描述
eui-cli publish
```

### 发布包的名字建议规范

- 前端研发部内部约定前端统一发布在 @epoint-fe 命名空间下，即 package.json 中的 name 为 "@epoint-fe/*", 如： "@epoint-fe/eui-icon" "@epoint-fe/eui-components" "@epoint-fe/eui-theme-idea"。

- 建议包安装功能或者相关业务，指定好统一的前缀：比如前端框架ui相关的全部以 "@epoint-fe/eui-" 开头， 电子表单相关的包，名称都以 "@epoint-fe/sform-" 开头。

### 包发布到仓库里，但是就是更新不到此包的最新版怎么办？
- 如果是针对vue-web工程，在工程根目录使用`eui-cli update`更新到最新即可
- 如果是针对工作空间内所有的包：
 1. 删除工作空间内的`pnpm-lock.yaml`
 2. 删除工作空间内的`node_modules`文件夹(推荐在idea里删，windows资源管理器中删太慢)
 3. 执行 `pnpm store prune`  清理掉本机 的pnpm缓存
 4. 执行`pnpm install --force` 后就是最新了
