# AI Coding Skills 体系设计：从 Prompt、上下文到可复用工作流

这两年很多团队都在积累 Prompt、常用问法和自动化脚本，但真正能沉淀为长期资产的，往往不是“某一句效果不错的提示词”，而是可以被稳定触发、稳定执行、稳定复盘的一套 `Skill`。

如果要做一个高质量的 AI 知识库，`Skill` 是非常值得优先建设的层。

## 什么是 Skill

在 AI 编码场景里，`Skill` 可以理解为一份面向模型的“能力说明书”，它至少应该回答下面几个问题：

- 什么时候触发这项能力
- 执行前需要读取哪些上下文
- 任务应该按什么步骤推进
- 产出应该长成什么样
- 如何判断这次执行是否合格

所以，`Skill` 和普通 Prompt 的区别不只是“写得更长”，而是它更接近一份可执行规范。

## 不要把这些概念混在一起

为了避免后面越做越乱，建议先把 5 个概念分清楚。

| 概念 | 作用 | 典型问题 |
| --- | --- | --- |
| Prompt | 单次对话指令 | 容易依赖上下文，不稳定 |
| 模板 | 固定表达骨架 | 适合批量写作，不一定能执行复杂任务 |
| Skill | 可复用能力说明 | 需要定义触发条件和验收标准 |
| Tool | 外部能力接口 | 解决“做不到”的问题，不解决“怎么做” |
| Workflow | 多步流程编排 | 解决复杂任务协作，但依赖 Skill 质量 |

很多团队推不动 AI 落地，不是因为工具不够，而是这些层次混在一起，导致“看起来什么都有，实际上没有一个能复用”。

## 一个好 Skill 应该长什么样

我更推荐把 Skill 拆成 5 层，而不是只写一大段说明。

### 1. 触发层

定义什么时候应该使用它。

例如：

- 用户要求“新增一个 VitePress 文章并更新侧边栏”
- 用户提到“请 review 当前改动，优先找风险”
- 用户想把 React 页面迁移到某个既有设计系统

这一层的目标，是提高召回准确率，避免 Skill 过于宽泛导致误触发。

### 2. 上下文层

定义执行前必须补齐哪些信息。

例如：

- 先读项目根目录说明
- 再看目标模块的入口文件
- 最后确认已有脚本和现有写作规范

很多 Skill 失败，不是执行步骤不对，而是上下文拿错了。模型一旦基于错误上下文开始推理，后面越努力越偏。

### 3. 执行层

把任务拆成几步可复现的动作。

好的执行层通常具备两个特征：

- 步骤数有限，便于模型遵循
- 每一步都有明确的完成信号

例如：

1. 扫描目标目录和现有内容，确认信息缺口  
2. 确定本次变更范围，避免无关改动  
3. 先改结构性文件，再补内容文件  
4. 运行构建或测试验证  
5. 汇总结果与后续建议

### 4. 输出层

明确结果应该如何表达。

例如：

- 最终回答先给结论，再列变更点
- Review 模式必须先给问题，再给总结
- 新增文章必须包含 frontmatter、结构化目录和小结

如果没有输出约束，模型即使把活做对了，结果也可能“不可交付”。

### 5. 校验层

给出最少但关键的验收标准。

例如：

- 是否真的运行过构建
- 是否引用了正确文件路径
- 是否修改了不该改的文件
- 文章结构是否完整，标签是否合理

这一层很重要，因为它让 Skill 从“会做”升级为“做得可追溯”。

## 一个最小可用 Skill 模板

下面这个骨架很适合团队先跑起来。

````markdown
# Skill Name

## 适用场景
- 用户要新增内容并同步导航
- 用户要求改动后完成基础验证

## 执行前检查
- 阅读项目说明与目录结构
- 确认目标文件是否已有未提交改动
- 识别是否需要运行脚本更新索引

## 执行步骤
1. 分析现有内容与命名方式
2. 在最小范围内修改目标文件
3. 生成或更新导航、索引、标签
4. 运行构建验证

## 输出要求
- 先说明完成情况
- 再说明新增内容和可继续优化点
- 标注未执行或执行失败的验证项

## 校验标准
- 导航可访问
- 新文章进入侧边栏
- 构建通过
````

这个模板不复杂，但已经能覆盖大多数内容型项目的基础场景。

## 评估 Skill 质量，重点看这 4 个指标

### 1. 召回准确率

触发时机是否对。一个过宽的 Skill 会抢走很多并不适合它的任务。

### 2. 完成稳定性

同类任务执行 10 次，是否能稳定覆盖关键步骤，而不是这次忘了校验、下次忘了读目录。

### 3. 上下文成本

为了用这个 Skill，模型是否要先读取过多无关文件。上下文越重，执行成本越高，稳定性也越差。

### 4. 结果可验证性

是否有明确检查点，例如“运行了什么命令”“改了哪些文件”“最终页面是否可构建”。

## Skill 什么时候值得单独沉淀

并不是每个问题都要上升成 Skill。下面几类最值得沉淀：

- 高频重复：一周会出现多次
- 结果有固定结构：例如 Review、脚手架、迁移、发布检查
- 对项目上下文依赖强：通用 Prompt 不够用
- 新人也会反复踩坑：适合通过 Skill 固化最佳实践

如果一个任务只会做一次，或者强依赖即时判断，先别着急抽 Skill。

## 团队落地时的建议顺序

### 第一阶段：先做“通用但刚需”的 Skill

优先沉淀：

- 代码审查
- 新增文章/页面
- Bug 排查
- 发布前检查

这类 Skill 覆盖面大，最容易让团队感受到收益。

### 第二阶段：再做“项目定制”的 Skill

例如：

- 某个组件库的接入规范
- 某个仓库的提交流程
- 某类接口文档的整理方式

这一步的目标，是把项目知识变成“模型可调用的组织资产”。

### 第三阶段：建立评估与淘汰机制

不是所有 Skill 都值得一直保留。建议按月回看：

- 使用频率
- 命中率
- 是否已经过时
- 是否与其他 Skill 重复

如果没有评估机制，Skill 库很快会变成另一个无人维护的文档目录。

## 常见失败模式

### Skill 写得太大

“所有前端问题都用这个 Skill” 基本等于没有 Skill。

### 只有方法，没有边界

只写“要认真分析、要注意质量”这类空话，模型执行时依旧无从下手。

### 把外部工具写成默认依赖

一旦工具不可用，整个 Skill 就失效。正确做法是写清楚降级路径。

### 没有输出契约

模型可能完成了任务，但结果格式不适合直接交付，团队依旧要做二次整理。

## `skills.sh` 适合怎么用

`skills.sh` 更像一个外部的 Agent Skills 目录和排行榜，适合做三件事：

- 看别人都在沉淀哪些高频能力
- 找可参考的 Skill 命名和拆分方式
- 观察哪些场景已经形成相对稳定的社区模式

但它不应该直接代替你自己的团队 Skill 库。因为真正决定效果的，往往是你们仓库的目录结构、交付规范、测试方式和沟通风格。

## 总结

如果把 Prompt 看作一次性的灵感，那么 Skill 更像是把灵感工程化、产品化、团队化的过程。

一个高质量 Skill 不追求“写得很厉害”，而追求三件事：

- 触发时机准
- 执行步骤稳
- 结果可验证

只要沿着这个方向持续沉淀，AI 在团队中的角色就会逐渐从“偶尔能帮忙的工具”变成“可协作的能力层”。
