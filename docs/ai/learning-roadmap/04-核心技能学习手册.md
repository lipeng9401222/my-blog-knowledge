# 04 核心技能学习手册

## 使用方式

这份文档不是课程列表，而是岗位能力地图。每个技能都按以下结构学习：

1. 学什么。
2. 学到什么程度。
3. 做什么练习。
4. 面试怎么讲。
5. 常见坑。

## 技能 1：Python 工程基础

### 为什么重要

岗位要求熟练 Python，并且要完成数据处理、模型训练、API 服务、脚本自动化。Python 是整条路线的地基。

### 必学内容

| 内容 | 学习目标 |
|---|---|
| 基础语法 | 变量、条件、循环、函数、列表、字典、集合 |
| 文件处理 | CSV、JSON、文本、Excel |
| 函数和模块 | 能拆分代码，避免全部写在一个脚本 |
| 异常处理 | 输入错误、文件不存在、接口失败时不崩溃 |
| 日志 | 用 logging 记录运行状态 |
| 虚拟环境 | venv、conda 或 uv 任一即可 |
| 依赖管理 | requirements.txt 或 pyproject.toml |
| Git | 提交、分支、README、忽略文件 |

### 岗位可用标准

你要能做到：

1. 写一个命令行脚本读取原始数据并输出清洗结果。
2. 把代码拆成读取、清洗、特征、训练、评估模块。
3. 关键步骤有日志。
4. 错误输入有明确报错。
5. README 能让别人跑起来。

### 练习任务

写一个 `industrial_data_cleaner` 小工具：

| 功能 | 要求 |
|---|---|
| 输入 | 一个 CSV 文件路径 |
| 输出 | 清洗后的 CSV 和数据质量报告 |
| 处理 | 缺失值、重复值、时间排序、异常值标记 |
| 日志 | 记录每一步处理了多少数据 |

### 面试表达

可以这样说：

> 我在项目里把数据处理流程拆成 reader、cleaner、feature_builder、reporter 几个模块，并用日志记录每一步的数据量变化。这样当工业数据出现缺失、时间乱序或字段异常时，可以快速定位是哪一步出了问题。

### 常见坑

| 坑 | 避免方式 |
|---|---|
| 所有代码写在 Notebook | 核心逻辑沉淀成 `.py` 模块 |
| API key 写死 | 使用环境变量或配置文件 |
| 只会跑通一次 | 写 README 和示例数据 |
| 没有日志 | 所有批处理脚本都加 logging |

## 技能 2：pandas/NumPy 数据处理

### 为什么重要

工业 AI 项目大量时间花在数据清洗、特征工程、标注和质量检查上。岗位明确写了 DCS/SCADA 时序数据处理。

### 必学内容

| 内容 | 学习目标 |
|---|---|
| DataFrame 基础 | 选择、过滤、排序、合并 |
| 分组聚合 | 按设备、班次、批次、日期聚合 |
| 缺失处理 | 删除、填充、插值、标记 |
| 异常处理 | 阈值、箱线图、3-sigma、业务规则 |
| 时间序列 | 时间戳解析、排序、重采样、滑窗 |
| 特征工程 | lag、rolling、diff、ratio、统计特征 |
| 数据导出 | CSV、Excel、Parquet |

### 岗位可用标准

你要能做到：

1. 读取 10 万行以上 CSV 不慌。
2. 按时间戳对齐多传感器数据。
3. 按 1 分钟、5 分钟、1 小时重采样。
4. 构造过去 N 分钟的滑窗特征。
5. 输出训练集、验证集、测试集。
6. 写数据质量报告。

### 练习任务

构造一份设备数据：

| 字段 | 示例 |
|---|---|
| timestamp | 2026-06-03 14:41:00 |
| equipment_id | PUMP_001 |
| temperature | 68.3 |
| pressure | 1.42 |
| vibration | 0.08 |
| current | 12.5 |
| status | normal/fault |

完成：

1. 处理缺失值。
2. 删除重复时间戳。
3. 标记温度过高、压力过低、振动突变。
4. 构造过去 5/15/30 分钟均值、最大值、标准差。
5. 输出模型训练表。

### 面试表达

> 工业时序数据清洗时，我会先检查时间戳连续性、点位缺失率、异常值分布和单位一致性。特征工程上会构造滑窗统计、滞后特征、变化率特征，并特别避免用未来数据造成标签泄漏。

### 常见坑

| 坑 | 避免方式 |
|---|---|
| 随机切分时序数据 | 按时间顺序切分 |
| 忘记排序 | 所有时序处理前先按时间排序 |
| 标签泄漏 | 构造特征时只使用预测时刻之前的数据 |
| 只删异常 | 工业异常可能正是故障信号，先标记再决定 |

## 技能 3：scikit-learn 传统机器学习

### 为什么重要

传统 ML 是工业 AI 的高性价比起点。很多工业数据量不大、特征结构明确，Random Forest、GBDT、线性模型常常比复杂深度模型更稳。

### 必学内容

| 内容 | 学习目标 |
|---|---|
| 训练/验证/测试 | 理解泛化能力 |
| 分类 | 故障识别、质量合格判断 |
| 回归 | 质量指标、能耗、产量预测 |
| 异常检测 | 设备异常、工况异常 |
| Pipeline | 标准化、特征处理、模型串联 |
| 指标 | Accuracy、Precision、Recall、F1、MAE、RMSE |
| 调参 | GridSearch、RandomizedSearch |
| 模型保存 | joblib 保存和加载 |

### 岗位可用标准

你要能做到：

1. 针对一个工业问题选择分类、回归或异常检测。
2. 建立 baseline。
3. 解释指标，尤其是误报和漏报。
4. 保存模型并在 API 中加载。
5. 写模型评估报告。

### 练习任务

做一个设备故障预测：

| 步骤 | 要求 |
|---|---|
| 数据 | 使用模拟或公开预测性维护数据 |
| 特征 | 滑窗均值、标准差、最大值、变化率 |
| 模型 | Logistic Regression、Random Forest、Gradient Boosting |
| 指标 | Precision、Recall、F1、混淆矩阵 |
| 输出 | 模型文件和评估报告 |

### 面试表达

> 对故障预测这类问题，我不会只看准确率，因为故障样本少，准确率容易虚高。我更关注召回率和漏报成本，同时结合误报率评估是否会造成报警疲劳。

### 常见坑

| 坑 | 避免方式 |
|---|---|
| 只训练一个模型 | 至少做 baseline 对比 |
| 指标单一 | 分类看混淆矩阵，回归看误差分布 |
| 类别不平衡忽略 | 使用 class_weight、重采样或调整阈值 |
| 调参过度 | 先保证数据和特征可靠 |

## 技能 4：PyTorch/TensorFlow 深度学习

### 为什么重要

岗位写了 PyTorch 或 TensorFlow，并且加分项提到 LSTM/Transformer 等工业时序预测。建议优先学 PyTorch，TensorFlow 了解即可。

### 必学内容

| 内容 | 学习目标 |
|---|---|
| Tensor | 理解张量和维度 |
| Dataset/DataLoader | 自定义时序数据集 |
| 训练循环 | forward、loss、backward、step |
| MLP | 基础神经网络 |
| LSTM | 序列预测 |
| Transformer | 了解 Attention 和 Encoder |
| 保存加载 | 模型权重保存和推理 |

### 岗位可用标准

你要能做到：

1. 用 PyTorch 训练一个简单模型。
2. 能讲清输入维度。
3. 能处理多变量时间序列。
4. 能比较深度模型和传统 ML。
5. 不盲目说深度模型一定更好。

### 练习任务

完成时序预测实验：

| 模型 | 任务 |
|---|---|
| Moving Average | 作为最简单 baseline |
| Random Forest | 传统 ML baseline |
| LSTM | 深度学习模型 |
| Transformer Encoder | 进阶对比 |

输出：

1. 训练曲线。
2. 测试集指标。
3. 预测值和真实值对比图。
4. 误差最大的时间段分析。

### 面试表达

> 我会先用移动平均、树模型等 baseline 建立下限，再尝试 LSTM 或 Transformer。工业数据量较小或噪声较大时，深度模型不一定优于传统模型，所以我会通过时间顺序切分和误差分析来判断是否值得使用复杂模型。

### 常见坑

| 坑 | 避免方式 |
|---|---|
| 维度搞混 | 明确 batch、seq_len、feature_dim |
| 数据泄漏 | 时序必须按时间切分 |
| 不做 baseline | 深度模型必须和简单方法比较 |
| 只看 loss | 输出业务指标和误差分析 |

## 技能 5：大模型 API 与提示词工程

### 为什么重要

岗位明确写了 DeepSeek/Qwen 等大模型应用。企业落地通常不是训练大模型，而是调用 API、设计提示词、接入业务系统。

### 必学内容

| 内容 | 学习目标 |
|---|---|
| API 调用 | 会传入 messages 并解析响应 |
| Prompt 结构 | system、user、上下文、输出格式 |
| 结构化输出 | JSON、表格、分类结果 |
| 参数 | temperature、max_tokens、top_p |
| 安全 | API key、敏感信息、注入风险 |
| 失败处理 | 超时、重试、限流、空响应 |

### 岗位可用标准

你要能做到：

1. 调用 DeepSeek 或 Qwen API。
2. 把工业文本分类或总结成结构化结果。
3. 控制输出格式。
4. 处理调用失败。
5. 不把 API key 写进代码仓库。

### 练习任务

报警文本分类：

输入：

```text
PUMP_001 轴承温度连续 10 分钟超过 85 摄氏度，振动值持续升高。
```

输出：

```json
{
  "level": "warning",
  "equipment": "PUMP_001",
  "reason": "轴承温度高且振动升高",
  "suggestion": "检查润滑状态和轴承磨损情况"
}
```

### 面试表达

> 我会把大模型当成一个可控的业务能力组件，而不是直接让它自由回答。比如报警分析场景中，我会通过 system prompt 限定角色和输出 JSON 结构，并对结果做字段校验和异常兜底。

### 常见坑

| 坑 | 避免方式 |
|---|---|
| 输出不稳定 | 使用结构化 prompt 和 JSON schema |
| 幻觉 | 对事实类问题接入 RAG |
| 泄露密钥 | 使用环境变量 |
| 没有重试 | API 调用封装超时和重试 |

## 技能 6：RAG 工业知识库

### 为什么重要

岗位明确写了私域知识库构建，包括规程、手册、案例文档的清洗、向量化和集成。

### 必学内容

| 内容 | 学习目标 |
|---|---|
| 文档加载 | PDF、Word、Markdown、HTML |
| 文档清洗 | 去噪、保留标题、表格处理 |
| 切分 | chunk size、overlap、按标题切分 |
| Embedding | 文本向量化 |
| 向量库 | Chroma、FAISS、Milvus 任一 |
| 检索 | top_k、相似度、metadata |
| 生成 | 基于检索内容回答 |
| 评测 | 检索命中率、回答正确率、幻觉案例 |

### 岗位可用标准

你要能做到：

1. 把规程文档清洗成结构化文本。
2. 建立向量索引。
3. 输入问题召回相关段落。
4. 基于段落生成答案。
5. 答案带引用来源。
6. 检索不到时不乱答。

### 练习任务

构建一个“设备维护规程知识库”：

| 功能 | 要求 |
|---|---|
| 文档入库 | 至少 5 份文档 |
| 检索 | 返回 top 3 相关片段 |
| 问答 | 回答必须基于检索内容 |
| 来源 | 显示文档名和段落编号 |
| 评测 | 准备 30 个问题 |

### 面试表达

> 我做 RAG 时会重点控制两件事：一是检索质量，二是答案可追溯。工业规程类问题不能让模型自由发挥，所以我会要求答案必须基于召回片段，并在检索不到依据时明确返回无法回答。

### 常见坑

| 坑 | 避免方式 |
|---|---|
| 文档没清洗 | 先做清洗和 metadata |
| chunk 太随意 | 按标题和语义边界切分 |
| 没有引用 | 每个 chunk 保留来源 |
| 不评测 | 准备固定问题集 |

## 技能 7：FastAPI/Flask 服务开发

### 为什么重要

岗位要求把模型封装成 API，对接 MES/工业互联网平台。FastAPI 是最推荐的主线。

### 必学内容

| 内容 | 学习目标 |
|---|---|
| 路由 | GET、POST |
| 请求体验证 | Pydantic model |
| 响应结构 | JSON、错误码 |
| Swagger | 自动接口文档 |
| 模型加载 | 启动时加载模型 |
| 日志 | 请求日志、耗时 |
| 异常处理 | 统一错误返回 |
| 配置 | 环境变量、配置文件 |

### 岗位可用标准

你要能做到：

1. 实现 `/health`。
2. 实现 `/predict`。
3. 实现 `/rag/query`。
4. 输出 Swagger 文档。
5. 写接口调用示例。
6. Docker 化部署。

### 练习任务

接口设计：

| 接口 | 作用 |
|---|---|
| `GET /health` | 服务健康检查 |
| `GET /version` | 返回模型和服务版本 |
| `POST /predict` | 输入传感器数据，返回风险 |
| `POST /rag/query` | 输入问题，返回知识库答案 |

### 面试表达

> 我会把模型能力封装为 RESTful API，接口层负责参数校验、日志、错误处理和响应格式，模型层只负责推理逻辑。这样后续对接 MES 或工业互联网平台时，调用方不需要关心模型内部实现。

### 常见坑

| 坑 | 避免方式 |
|---|---|
| 接口没有校验 | 使用 Pydantic |
| 模型每次请求都加载 | 启动时加载，全局复用 |
| 没有错误码 | 统一异常处理 |
| 没有文档 | Swagger 和 README 都要写 |

## 技能 8：Docker 部署

### 为什么重要

岗位要求 Docker 基本操作。对 AI 应用工程师来说，Docker 是把服务交给别人运行的最低工程能力。

### 必学内容

| 内容 | 学习目标 |
|---|---|
| 镜像 | 理解 image |
| 容器 | 理解 container |
| Dockerfile | 会写基础 Dockerfile |
| 端口映射 | `-p 8000:8000` |
| 环境变量 | 传 API key 和配置 |
| 日志 | 查看容器日志 |
| docker compose | 可选，多个服务时使用 |

### 岗位可用标准

你要能做到：

1. 为 FastAPI 项目写 Dockerfile。
2. 构建镜像。
3. 运行容器。
4. 访问容器内接口。
5. 用环境变量传配置。

### 练习任务

为模型服务写 Dockerfile：

```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### 面试表达

> 我会用 Docker 固化 Python 版本、依赖和启动命令，避免在不同机器上因为环境差异导致服务跑不起来。部署时通过环境变量注入模型路径、API key 和日志级别。

### 常见坑

| 坑 | 避免方式 |
|---|---|
| 镜像太大 | 使用 slim 基础镜像 |
| 密钥写进镜像 | 用环境变量 |
| 端口不通 | 检查 host、port 和端口映射 |
| 依赖不锁定 | requirements 固定版本 |

## 技能 9：工业系统和场景理解

### 为什么重要

这个岗位有明显工业属性。即使不是工艺专家，也要能听懂业务人员在说什么。

### 必学内容

| 概念 | 学习目标 |
|---|---|
| DCS | 分布式控制系统，偏过程控制 |
| SCADA | 监控与数据采集，偏监控和采集 |
| MES | 制造执行系统，偏生产计划、工单、批次、质量 |
| 工业互联网平台 | 设备接入、数据汇聚、应用开发 |
| 点位 | 一个传感器或控制变量 |
| 报警 | 超阈值、设备异常、状态变化 |
| 工单 | 运维和生产任务记录 |
| 批次 | 生产过程中的批量追踪单位 |

### 岗位可用标准

你要能做到：

1. 解释 MES、SCADA、DCS 的区别。
2. 画出工业 AI 数据流。
3. 设计一个传感器数据表。
4. 设计一个报警记录表。
5. 说明模型如何接入业务系统。

### 练习任务

画一张数据流：

```text
设备/传感器 -> DCS/SCADA -> 时序数据库 -> 数据清洗 -> 模型预测 -> FastAPI -> MES/工业平台 -> 运维人员
```

### 面试表达

> 我理解工业 AI 落地不是只训练模型，而是要接入现有工业系统。DCS/SCADA 负责采集和监控，MES 更偏生产执行和业务管理，模型服务通常通过 API 或消息方式把预测结果返回给平台，再由业务规则决定是否触发报警或工单。

### 常见坑

| 坑 | 避免方式 |
|---|---|
| 把 MES/SCADA/DCS 混为一谈 | 先记住职责差异 |
| 只讲模型不讲系统 | 每个项目都画数据流 |
| 不考虑误报 | 报警要结合业务阈值 |
| 不写文档 | 工业项目交付必须有文档 |
