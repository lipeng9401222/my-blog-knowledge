## 一、AI 系统架构与 Agent 原理

**Q1: AI xx 系统主要是做什么的？** [P0] *(ML公司)*

这是一道开场题，考的是你能否清晰描述自己构建的 AI 系统。回答框架：目标用户是谁 → 解决什么痛点 → 核心功能模块 → 技术架构分层。例如："系统面向工艺工程师，解决设备故障排查效率低的问题。核心功能包括传感器数据实时分析、故障自动分类、知识库问答、诊断报告生成。架构分四层：数据接入层（DCS/SCADA 数据采集）→ AI 服务层（模型预测 + RAG 问答）→ API 网关层（FastAPI）→ 前端展示层（Vue3）。"

**Q2: 有没有了解过像 Manus 这种通用 Agent？** [P1] *(ML公司)*

Manus 是 2025 年发布的通用型 AI Agent，能自主完成浏览网页、操作电脑、编写代码、处理文件等复杂任务。它与 ChatGPT 的区别是：ChatGPT 是"对话助手"（只输出文本建议），Manus 是"行动助手"（直接替你操作）。了解概念即可，重点说"通用 Agent 的核心是给 LLM 配备了浏览器、终端、文件系统等工具，让它能感知和操作数字环境"。

**Q3: 这些通用 Agent 一般分几层功能？** [P1] *(ML公司)*

通常分四层：感知层（接收用户指令/环境信息）→ 规划层（LLM 分解目标为子任务）→ 行动层（调用工具执行：浏览器/终端/代码执行/API 调用）→ 记忆层（短期对话记忆 + 长期向量记忆）。有些还加一个验证层（检查任务完成质量）。对应到 LangChain 就是：Input → Agent(ReAct) → Tools → Memory。

**Q4: 有没有构建过多智能体（Multi-Agent）架构？** [P0] *(ML公司)*

如果你研究过 Nexbot 项目，可以说："我研究过 Nexbot 的 Agent OS 多专家协作运行时。它用 Mission（协作边界）、Job（执行边界）、ContextPack（上下文边界）三个核心对象管理多 Agent 协作。Agent 之间通过 Mailbox 定向通信，通过 Blackboard 共享事实和决策。这比我用 CrewAI 做的简单角色分工更完善。"如果没有实操经验，至少能说出多 Agent 的概念和架构思路。

**Q5: 什么是多 Agent？** [P0] *(ML公司)*

多 Agent 是指多个有不同角色或能力的 LLM Agent 协同完成一个复杂任务。每个 Agent 有自己的 System Prompt（人格/专长）、工具集和记忆，它们通过消息传递或共享状态协作。类比人类团队：一个负责调研、一个负责分析、一个负责写报告，各司其职。

**Q6: 为什么需要多 Agent？** [P0] *(ML公司)*

三个核心原因：一是上下文隔离——每个 Agent 只需关注自己领域的上下文，避免单一 Agent 的上下文窗口被撑爆；二是工具聚焦——每个 Agent 只装自己需要的工具，减少模型的选择困难；三是专业化——每个 Agent 可以用不同的 System Prompt 和模型（如一个用 GPT-4 做推理，一个用 Qwen 做中文生成）。

**Q7: 什么场景下需要多 Agent 架构？** [P0] *(ML公司)*

当任务有明确的角色分工且各角色需要不同工具/知识时。典型场景：代码开发（需求分析+编码+测试+Review 各一个 Agent）、研究报告（调研+分析+写作）、客服系统（意图识别+业务处理+质量审核）。如果任务简单到单个 Agent 几步就能完成，不需要多 Agent。

**Q8: 单个 Agent 其实也可以分配多个职能，为什么还要多 Agent？** [P1] *(ML公司)*

单 Agent 多职能的问题：上下文越长模型注意力越分散（"Lost in the Middle"问题）、工具太多模型选择准确率下降、一个角色出错会连锁影响其他角色、难以并行执行。多 Agent 各自上下文干净、工具聚焦、可并行、错误隔离。但多 Agent 也有代价——通信开销增大、协调逻辑复杂、Token 总消耗可能更高。

**Q9: 多 Agent 主要是为了解决什么问题？** [P1] *(ML公司)*

核心解决三个问题：上下文窗口限制（单 Agent 无法容纳所有信息）、工具选择困难（单 Agent 面对几十个工具时选择准确率下降）、专业化不足（一个 Prompt 很难让模型同时擅长多个领域）。多 Agent 通过"分而治之"思路，让每个 Agent 专注一个领域。

**Q10: 介绍一下你的 AI 项目的架构情况？** [P0] *(JR公司)*

回答框架（参照 Nexbot 架构）：入口层（HTTP/SSE/WebSocket）→ 网关层（鉴权/限流/幂等）→ 服务层（AgentService 门面，聚合资源/上下文/运行时/通道四个领域）→ 运行时层（LangGraph 状态图：oracle→action→terminal 循环）→ 存储层（文件仓库/SQL + 向量库）。关键强调分层解耦和工具系统设计。

**Q11: 抽象通用 AI 能力模块做了什么？** [P1] *(JR公司)*

指把 AI 能力下沉为可复用的基础模块。典型包括：模型调用抽象（统一 OpenAI/Anthropic/Qwen 接口）、Prompt 管理抽象（模板化+版本管理）、工具注册抽象（统一 Tool 注册和发现）、RAG 管道抽象（文档加载→切分→向量化→检索→生成标准化流程）、记忆抽象（对话历史管理接口）。目的是让上层业务只关心业务逻辑，不关心 AI 底层。

**Q12: AI 调用工具失败了怎么解决？** [P0] *(JR公司)*

四层策略：工具内部 try-except，返回结构化错误信息给 Agent（不是抛异常中断）；Agent 收到错误后能自行调整策略（换个参数重试或换一个工具）；设置最大重试次数防止死循环；关键工具加降级方案（如 API 不可用时用本地缓存）。在 LangChain 中，工具返回错误信息作为 Observation 传给 Agent，Agent 会在下一步 Thought 中分析错误并决定重试或换策略。

**Q13: 你这个 AI XX 系统，如果从架构角度看，分成哪些层？** [P0] *(YC公司)*

参照 Nexbot 五层架构回答：入口层（多种请求入口）→ 控制面（鉴权/准入/限流/流式编排）→ 服务门面（AgentService 聚合四大领域）→ 运行时引擎（LangGraph 图：模型决策→工具执行→路由循环）→ 存储层（文件/SQL/向量库/日志）。强调"Gateway 唯一入口"和"模型输入四层分层"两个核心设计。

**Q14: 假设现在让你做一个 xxx 门店的 AI 数字化系统，你会怎么设计？** [P0] *(YC公司)*

回答思路：先分析需求（门店有哪些 AI 场景：智能客服/库存预测/销售分析/员工培训）→ 技术选型（FastAPI 后端 + Vue 前端 + Qwen 大模型 + RAG 知识库 + MySQL 数据库）→ 架构分层（数据采集→AI 服务→API 网关→前端）→ 落地优先级（先做 RAG 客服，再做销售分析）。关键展示系统设计思维。

**Q15: 技术架构：你怎么拆模块？数据怎么流转？技术选型怎么做？** [P0] *(YC公司)*

拆模块按职责：数据模块（采集/清洗/存储）、AI 模块（模型调用/RAG/Agent）、API 模块（路由/鉴权/限流）、前端模块（UI/状态管理/流式渲染）。数据流转：传感器/业务数据 → 数据清洗 → 特征提取 → 模型推理/RAG 检索 → 结果通过 SSE 推送前端。选型原则：成熟稳定优先、社区活跃度、团队熟悉度、成本可控。

**Q16: 如何推动 AI 系统落地的？** [P1] *(PY公司)*

回答框架：找到真实痛点（不要为了 AI 而 AI）→ 做 MVP 原型验证可行性 → 找种子用户试用收集反馈 → 量化效果（效率提升 X%）→ 逐步推广。关键点：先解决一个具体问题（如"设备故障查询慢"），不要一上来就做大而全的 AI 平台。

**Q17: 用户问题如何路由到 AI 系统？** [P1] *(PY公司)*

两种方式：一是基于规则的路由（关键词匹配/正则，如包含"故障"走 AI 诊断，包含"库存"走传统查询）；二是基于意图识别的路由（用小模型或 LLM 分类用户意图，决定走 AI 还是传统接口）。生产中通常先规则后 AI——规则覆盖高频场景，AI 兜底长尾问题。

**Q18: 意图识别是如何实现的？使用了哪些模型？** [P1] *(PY公司)*

意图识别有三种实现：基于关键词/正则（最快但最死板）、基于传统 ML 分类（TF-IDF + SVM/随机森林，速度快）、基于 LLM（Few-shot Prompt 让大模型分类，最灵活但有延迟）。生产中常用"规则 + LLM 兜底"的混合策略。模型可以选小模型（如 BERT 微调）做快速分类，也可以直接用 Qwen/DeepSeek 做 Few-shot。

**Q19: 如何处理未登录用户的问题查询？** [P2] *(PY公司)*

未登录用户的处理：限制访问频率（IP 维度限流）、限制功能范围（只开放公开知识库查询，不开放敏感数据）、会话临时 ID（用 session/cookie 追踪但不持久化）、引导注册（免费试用 N 次后要求登录）。技术上用 Redis 做临时会话管理。

**Q20: AI 输出如何保证序列化/结构化？为什么选择 Markdown 而不是 JSON？** [P0] *(PY公司)*

结构化输出靠三种方式：Function Calling/Tool Use（模型原生支持，最可靠）、JSON Mode（部分模型支持，强制输出合法 JSON）、Prompt 指令 + OutputParser（最通用但不可靠）。选 Markdown 而非 JSON 的原因：Markdown 对人类可读性更好（直接渲染为富文本），模型生成 Markdown 比生成 JSON 更自然（训练数据中 Markdown 更多），容错性好（少个标记不会整体解析失败）。JSON 适合机器间传递，Markdown 适合人机交互。

**Q21: 简单简述下你的 AI XX 系统用途、用到哪些技术？** [P0] *(YF公司)*

这是开场题，用一句话概括 + 技术栈列表。例如："系统用于工业设备智能诊断，整合传感器数据分析和知识库问答。技术栈：Python + FastAPI + LangChain + RAG + Qwen/DeepSeek + Chroma 向量库 + Docker 部署。"

**Q22: 如果从零搭建一套已选定大模型的通用型智能客服，说下整体技术选型与落地思路？** [P0] *(YF公司)*

落地思路：知识库构建（收集 FAQ/产品文档/历史工单 → 清洗 → 切分 → 向量化 → 入库）→ 意图路由（区分闲聊/知识库查询/业务办理）→ RAG 问答（检索 + 生成 + 引用溯源）→ 多轮对话管理（Memory）→ 人工兜底（AI 把握度低时转人工）。技术选型：FastAPI + LangChain + 向量库(Chroma/Milvus) + Redis(会话缓存) + Docker。

## 二、LangChain / LangGraph / Agent 框架

**Q23: LangChain / LangGraph 主要用了什么 AI 技术？** [P0] *(ML公司)*

LangChain 核心技术：LCEL 链式调用（Prompt | Model | Parser 管道）、Agent + Tools（ReAct 循环自主工具调用）、Memory（对话记忆管理）、Retriever（RAG 检索抽象）。LangGraph 核心技术：StateGraph（有状态有向图）、条件路由（动态选择下一节点）、interrupt（人机交互中断恢复）、checkpointing（状态持久化和恢复）。

**Q24: 为啥选择 LangChain、LangGraph？选型的依据是啥？** [P0] *(JR公司)*

选型理由：LangChain 生态最成熟、社区最大、文档最全、组件最丰富（兼容几乎所有 LLM 和工具），降低学习成本和招聘难度。LangGraph 解决了 AgentExecutor 的"黑盒循环"问题——当 Agent 流程复杂（需要条件分支、人机交互、并行执行）时，显式图编排比黑盒循环更可控、更可调试。替代方案 LlamaIndex 更专注 RAG 但 Agent 能力弱，AutoGen/CrewAI 更适合多 Agent 但生态小。

**Q25: LangChain 不同版本/模块的区别？** [P1] *(HQ公司)*

LangChain 经历了从 `langchain`（一体包）到 `langchain-core` + `langchain` + `langchain-community` + `langchain-experimental` 的拆分。核心包 `langchain-core` 提供 Runnable、PromptTemplate 等基础抽象；`langchain` 提供 Chain/Agent/Memory 等高级 API；`langchain-community` 是第三方集成（各 LLM/向量库/工具的适配器）；`langchain-experimental` 是实验性功能。拆分的好处是按需安装、减少依赖冲突。

**Q26: 多智能体中动态路由如何根据条件分配任务分支？** [P1] *(HQ公司)*

两种方式：基于 LLM 的路由（用一个轻量 LLM 调用做分类决策，输出"该交给哪个 Agent"）和基于规则的路由（关键词/正则匹配 + 优先级队列）。LangGraph 中用 `add_conditional_edges` 实现——节点输出一个字符串（如 "research"/"analyze"/"write"），条件边根据该字符串路由到对应节点。CrewAI 中用 `process="hierarchical"` 由 manager Agent 做分配。

**Q27: 说说 MCP、Function、Skills 三者的区别？** [P1] *(YF公司)*

Function Calling 是模型原生的工具调用能力——在请求中声明函数 schema，模型决定何时调用。MCP（Model Context Protocol）是 Anthropic 提出的标准化工具协议——把工具、资源、Prompt 统一为标准接口，任何 MCP 兼容的客户端都能用。Skills 是更高层的封装——一个 Skill 是"Prompt + 工具 + 知识"的打包，模型按需加载。层次关系：Function 是底层机制，MCP 是协议标准，Skills 是应用层封装。

**Q28: 有在使用什么 MCP 或者 SKILL？** [P2] *(XZ公司)*

如果你用 QoderWork/Claude Code 等工具，可以提到："我使用过 MCP 协议的工具集成——在 Nexbot 项目中，MCP 工具是 DynamicToolLoader 的八种工具来源之一，通过 stdio/sse 接入外部 MCP server。Skills 是打包好的 Prompt+工具+知识组合，在 Nexbot 中有完整的 Skill 注册和加载机制（skill_discover + skill_load 工具）。"

**Q29: 项目中是否自主设计过智能体？具体实现方式是什么？** [P0] *(YU公司)*

如果有实操经验，讲你的 Agent 设计：System Prompt 设计（角色/约束/输出格式）、工具集定义（哪些工具 + 各自的 schema）、Agent 类型选择（ReAct vs OpenAI Tools）、Memory 策略（Buffer vs Summary）、错误处理和迭代限制。如果没实操过，用 Nexbot 项目参照："我研究过 Nexbot 的 Agent 设计——oracle 节点用模型做决策，action 节点执行工具，通过条件路由决定循环还是终止。工具通过 DynamicToolLoader 按权限和入口点动态加载。"

**Q30: 有没有了解之前 claude code 泄露的事件？有没有研究过源码？** [P2] *(JR公司)*

Claude Code 的"泄露"是指其 System Prompt 被用户通过特定技巧提取出来并公开。研究过的人可以提到：Claude Code 的 System Prompt 包含详细的工具使用规则、代码规范、安全约束等，是一个精心设计的 Agent Prompt 范例。面试时说"我了解过 Claude Code 的 Prompt 设计思路——它的分层工具定义、安全约束、和自主决策逻辑，对我的 Agent Prompt 设计有参考价值"即可。

**Q31: claude code 的核心循环流程说一下？** [P1] *(JR公司)*

Claude Code 的核心循环：接收用户指令 → LLM 分析指令决定下一步（思考/读文件/写文件/执行命令/搜索）→ 执行工具调用 → 观察结果 → 继续推理或返回结果。本质就是 ReAct 循环——Thought → Action → Observation → 循环直到完成。与 Nexbot 的 oracle→action 循环是同一个模式。

**Q32: claude code 的多层架构是怎么做的？** [P1] *(KJ公司)*

Claude Code 的架构（基于公开信息推断）：顶层是 CLI 交互层（接收用户输入、展示输出）；中间是 Agent 编排层（LLM 决策 + 工具调用循环，包含文件读写/代码执行/搜索等工具）；底层是工具执行层（实际的文件系统操作、代码运行、API 调用）。还有上下文管理层（管理对话历史和上下文窗口）和安全沙箱层（限制工具执行权限）。

**Q33: claude code 的上下文治理是怎么做的？** [P1] *(JR公司)*

Claude Code 的上下文治理包括：对话历史裁剪（保留最近 N 轮 + 保留关键工具调用结果）、文件内容压缩（大文件只取关键片段）、自动摘要（长对话用 LLM 总结旧内容）、上下文窗口监控（接近上限时主动压缩）。对应 Nexbot 架构文档 3.6 节的 RuntimeContextPipeline——5 个 Transformer 串联过滤消息（删空/按轮次过滤/保留最近 12 条工具消息/保留最近 48 条总消息）。

**Q34: claude code 上下文压缩时，如果大模型能力问题压缩质量有问题怎么解决？** [P1] *(JR公司)**

如果压缩模型能力不足导致摘要丢失关键信息：用更强的模型做摘要（如用 GPT-4 做摘要，用 GPT-3.5 做对话）；分段摘要再合并（每次只摘要一小段，避免模型丢失信息）；保留关键实体的"锚点"（明确告诉模型"必须保留这些关键词"）；人工设置压缩规则（基于规则的裁剪优先于 LLM 摘要，如"始终保留最后 5 轮对话原文"）。

**Q35: ps 网页版实现方案？canvas 无限画布是怎么实现的？** [P2] *(KJ公司)*

Photoshop 网页版的实现涉及 WebAssembly（把 C++ 编译为 WASM 在浏览器运行）、WebGL（GPU 加速渲染）、OffscreenCanvas（后台渲染不阻塞主线程）、虚拟 DOM + Canvas 混合渲染。无限画布的实现：用"视口+偏移量"计算可见区域，只渲染视口内的内容；缩放时调整渲染分辨率；大图用瓦片（tile）分割按需加载。这些是前端图形学领域知识，AI 工程师岗位了解概念即可。

**Q36: WebAssembly 有了解过吗？** [P2] *(KJ公司)*

WebAssembly（WASM）是一种浏览器可执行的二进制指令格式，让 C/C++/Rust 等语言编译后在浏览器运行。特点：接近原生性能（比 JS 快 5-20 倍）、跨平台、安全沙箱。在 AI 领域的用途：浏览器端跑推理（如 ONNX Runtime Web 用 WASM 在浏览器跑模型）、性能瓶颈计算用 WASM 加速。了解概念即可。

**Q37: express 的底层原理？如果让你设计实现怎么做？** [P2] *(KJ公司)*

Express.js 的底层原理：基于 Node.js http 模块，维护一个中间件数组（函数队列），请求到来时依次执行中间件（每个中间件可以修改 req/res 或调用 next() 传给下一个）。路由是基于 HTTP method + path 模式匹配。如果自己设计：实现中间件管道（洋葱模型或线性管道）、路由匹配（正则或前缀树）、请求/响应封装、错误处理中间件。NestJS（后面会问）就是在 Express 之上加了 IoC 容器和装饰器。

## 三、RAG 检索增强生成

**Q38: 简单介绍一下 RAG 的流程？** [P0] *(JR公司/YF公司)*

RAG 全流程分五步：文档加载（PDF/Word/HTML → Document 对象）→ 文档切分（按字符数/标题/语义切分为 Chunk）→ 向量化（用 Embedding 模型把 Chunk 转为向量）→ 存储（向量写入向量库）→ 检索生成（用户问题向量化 → 在向量库做相似度搜索 → 取 Top-K 上下文 → 拼入 Prompt → LLM 生成回答）。

**Q39: 怎么加速 RAG 的检索过程，有啥策略？** [P0] *(JR公司)*

加速策略：索引优化（HNSW 图索引比暴力搜索快几个数量级）、向量维度压缩（从 1536 维降到 768 维，速度提升一倍）、缓存热点查询（相同问题直接返回缓存结果）、预计算查询向量（高频问题预嵌入）、异步检索（流式返回，先返回检索结果再流式生成）、分布式检索（Milvus 等支持分片并行）。

**Q40: RAG 的检索是怎么做的？** [P0] *(TK公司)*

检索流程：用户问题 → Embedding 模型转为向量 → 在向量库做相似度搜索（默认余弦相似度或内积）→ 取 Top-K（通常 3-5 个 Chunk）→ 可选 Reranker 重排序 → 把检索到的 Chunk 拼入 Prompt 上下文 → 发给 LLM 生成回答。进阶：混合检索（向量 + 关键词 BM25）、Query 改写（把口语化问题改写为检索友好的查询）。

**Q41: 怎么确认召回的数据准确性？** [P0] *(JR公司，政策文件场景)*

政策文件场景对准确性要求极高，四种验证方式：引用溯源（回答中标注来源段落，人工可核查）、置信度过滤（检索分数低于阈值的 Chunk 不用）、多路召回投票（用不同检索方式各召回一批，取交集）、LLM 自我验证（让另一个 LLM 判断"回答是否被上下文支持"）。RAGAS 评估框架的 faithfulness 指标专门做这件事。

**Q42: 什么是查询改写？作用是什么？** [P0] *(HQ公司)*

查询改写（Query Rewriting）是在检索前把用户的原始问题转换为更适合检索的查询。作用：用户的提问往往是口语化的（如"那个设备老是报警怎么搞"），直接检索效果差。改写后变成"设备频繁报警 故障排查 方法"，检索更精准。进阶有 HyDE（让 LLM 先生成一个假设的回答，用回答去检索，因为回答和文档的语义更接近）。

**Q43: 你们上下文压缩的策略是什么？** [P0] *(JR公司)*

上下文压缩策略：文档级压缩（检索到的 Chunk 如果太长，用 LLM 摘要为关键句）、冗余过滤（去重相似度高的 Chunk）、上下文窗口限制（按 Token 预算裁剪，保留最相关的）、对话历史压缩（旧对话用 Summary Memory，近期用 Buffer Memory）。对应 Nexbot 的 token_budget.py 机制——每个大文本字段有 Token 上限裁剪。

**Q44: Elasticsearch + 向量库怎么做混合检索？整体流程？** [P1] *(HQ公司)*

混合检索流程：用户问题同时走两条路——向量检索（问题向量化 → 向量库搜索 Top-K）和关键词检索（问题分词 → ES BM25 搜索 Top-K）→ 合并两路结果 → 去重（相同文档只保留一次）→ 重排序（Reranker 模型对合并结果重新打分排序）→ 取最终 Top-K 作为上下文。ES 本身也支持向量检索（dense_vector 字段），可以不用单独向量库。

**Q45: 混合检索已经做过分组、排序、去重，为什么还需要再次精排（Re-rank）？** [P1] *(HQ公司)*

因为初始检索的排序基于相似度分数，但相似度高不等于回答质量高。向量检索可能召回"语义相似但答非所问"的 Chunk，关键词检索可能召回"包含关键词但上下文不相关"的 Chunk。Reranker 模型（如 bge-reranker、Cohere Rerank）用 Cross-Encoder 架构（而不是双塔 Bi-Encoder），能同时看到 Query 和 Document 做交叉注意力，判断"这个文档是否真的能回答这个问题"，精度远高于初始检索的相似度分数。

**Q46: 项目中有使用领域词典吗？** [P1] *(HQ公司)*

领域词典在工业 AI 中很重要：用于分词优化（工业术语如"板式换热器""DCS 控制器"不应被拆分）、同义词扩展（"泵"和"水泵"是同义词）、实体识别（识别文档中的设备名/参数名/故障类型）。实现方式：自定义词典注入 ES 的 IK 分词器、用领域词典做 Query 扩展（"泵故障" → 也检索"水泵故障""离心泵故障"）。

**Q47: 向量库目前的数据量？** [P1] *(JR公司)*

这是问你有没有处理过大规模数据。诚实回答。如果量小可以说"目前知识库约 X 万条 Chunk，用的是 Chroma 轻量级向量库"。如果量大可以说"百万级 Chunk，用 Milvus 分布式部署，HNSW 索引，查询延迟 < 50ms"。关键是要知道不同量级的选型：万级用 Chroma/FAISS，十万到百万用 Qdrant/Milvus，亿级用 Milvus 集群。

**Q48: 向量库使用的是啥？选型依据？** [P0] *(JR公司/HQ公司/YF公司)*

选型依据：数据规模（小规模用 Chroma 内嵌，大规模用 Milvus/Qdrant 分布式）、性能要求（FAISS 最快但是库不是服务，Milvus 持续维护 HNSW 图索引）、部署复杂度（Chroma 零配置，Milvus 需要 etcd+MinIO+Milvus 三件套）、生态兼容（是否支持 LangChain/LlamaIndex 集成）。常见选择：原型用 Chroma，生产用 Milvus 或 Qdrant。

**Q49: 你们知识库使用什么向量库？** [P0] *(HQ公司)*

同上。建议回答："原型阶段用 Chroma（轻量零配置），如果数据量大或需要生产部署，会选 Milvus（支持分布式、HNSW 索引、多租户）或 Qdrant（Rust 写的性能好、API 简洁）。"

**Q50: 怎么做向量化和知识库检索？** [P0] *(PY公司)*

向量化流程：文档切分为 Chunk → 调用 Embedding API（如 bge-m3、text-embedding-3）→ 得到向量 → 连同原文和元数据写入向量库。检索流程：用户问题 → 同一个 Embedding 模型向量化 → 向量库相似度搜索 → Top-K 结果。关键点：查询和入库必须用同一个 Embedding 模型，否则向量空间不一致。

**Q51: 文档、PDF、课件是如何切分、入库、人工审核的？** [P0] *(PY公司)*

切分策略：按字符数（RecursiveCharacterTextSplitter，chunk_size=500-1000，overlap=50-100）、按标题（MarkdownHeaderTextSplitter）、按语义（语义边界切分）。入库流程：切分 → 向量化 → 写入向量库 → 建立原文映射（可追溯）。人工审核：入库前预览 Chunk 质量（切分是否破坏语义）、标注 Chunk 的类别/权限级别、审核通过后正式入库。为什么要人工审核：自动切分可能把表格/公式/步骤说明切断，导致检索到的 Chunk 缺失关键上下文。

**Q52: 知识库入库并非全自动，为啥需要人工审核两道环节？** [P1] *(YF公司)*

两道审核的原因：第一道审内容质量（切分是否合理、内容是否准确、是否包含敏感信息）；第二道审检索质量（用预期问题测试检索是否能命中、Chunk 是否包含足够上下文让 LLM 生成正确回答）。AI 生成的内容入库后再被 RAG 检索出来给用户，如果原始内容有误会产生"错误放大"效应，所以需要人工把关。

**Q53: 知识库字幕总结这块是你负责搭建的吗？说下整体搭建思路与实现过程？** [P1] *(YF公司)*

如果是视频字幕场景：字幕提取（从视频/音频用 Whisper 等模型提取文字）→ 字幕清洗（去重复、修正标点、分段）→ 字幕摘要（用 LLM 按场景/主题分段总结）→ 向量化入库（把摘要和原始字幕都向量化）→ 检索问答（用户问视频中讲了什么 → 检索字幕摘要 → LLM 生成回答）。关键点：字幕有时间戳元数据，检索结果可以定位到视频具体位置。

**Q54: ES 的切片方式？ES 有哪些分词机制？** [P1] *(YF公司)*

ES 的分词机制：Standard（默认，按 Unicode 文本分割）、Simple（按字母数字分割）、Whitespace（按空格分割）、Keyword（不分词整体作为一个 token）、IK（中文分词器，支持 ik_smart 粗粒度和 ik_max_word 细粒度）。"切片方式"可能指 ES 的分片（shard）——索引分为多个分片分布在不同节点，支持水平扩展。每个分片是一个独立的 Lucene 索引。

**Q55: AI 回答效果差的时候，你会从哪些环节调整优化？** [P0] *(YF公司)*

排查链路：检索环节（检索到的 Chunk 是否相关？不相关 → 调整切分策略/换 Embedding 模型/加 Query 改写）、上下文环节（上下文是否足够？Token 预算是否太小导致截断？）、Prompt 环节（System Prompt 是否清晰？是否需要 Few-shot 示例？）、模型环节（模型能力是否够？换更强模型或调温度参数）、数据环节（原始文档质量是否有问题？）。从后往前逐个排查。

**Q56: Prompt 提示词是你编写还是专人负责？** [P0] *(YF公司)*

面试官想了解你的参与深度。最佳回答："Prompt 由我主导设计和迭代。我会用系统化的方法——定义角色、约束、输出格式、Few-shot 示例，然后通过 A/B 测试对比不同 Prompt 的效果。同时我也关注业务方的反馈，持续优化。"

**Q57: 你会怎么撰写需求，交给 AI 生成提示词？** [P1] *(YF公司)*

需求到 Prompt 的转化方法：先写自然语言需求描述（"我需要一个分析传感器数据的助手，输入是温度/压力/流量数据，输出是故障类型和维修建议"）→ 再转化为 Prompt 结构（角色定义 + 输入格式 + 输出格式 + 约束条件 + Few-shot 示例）→ 测试迭代（用真实数据测试，根据输出质量调整 Prompt）。

## 四、大模型选型与成本优化

**Q58: 大模型的选型逻辑？** [P0] *(JR公司)*

选型维度：能力（推理/代码/中文/多模态，看 benchmark 和实际测试）、成本（输入/输出 Token 价格，Qwen/DeepSeek 比 GPT-4 便宜 10 倍）、延迟（首 Token 延迟影响用户体验，流式输出缓解）、隐私（工业数据敏感，可能需要私有化部署）、上下文窗口（128K vs 32K，长文档 RAG 需要大窗口）。策略：高强度推理用强模型（GPT-4/Claude），日常对话用便宜模型（Qwen/DeepSeek），敏感数据用本地部署（Ollama + Qwen）。

**Q59: 大模型有没有做过微调？是直接接入第三方 API 还是私有化部署？** [P0] *(YF公司)*

如果没有微调经验，诚实回答："我们主要用 API 接入（Qwen/DeepSeek），没有做微调。原因：RAG + Prompt Engineering 已经能满足大部分需求，微调成本高（数据标注 + GPU 训练 + 评估）且迭代慢。如果遇到 RAG 解决不了的场景（如特定行业术语理解、特定输出风格），会考虑 LoRA 轻量微调。"这个回答展示你知道什么时候该微调、什么时候不该。

**Q60: 有没有考虑过怎么节约 AI 成本？** [P0] *(ML公司)*

成本节约策略：模型分级（简单任务用小模型，复杂任务用大模型，路由层做分类）、Prompt 精简（减少冗余描述、压缩上下文）、缓存（相同问题直接返回缓存结果，相似问题用语义缓存）、批量处理（Batch API 比实时 API 便宜 50%）、上下文裁剪（只传必要的历史消息）、本地部署高频调用（Ollama 跑小模型处理高频简单任务）。

**Q61: 做了哪些成本优化方案？** [P0] *(ML公司)*

具体方案：对话历史压缩（用 Summary Memory 替代 Buffer Memory，Token 降 70%）、RAG 上下文限制（Top-K 从 10 降到 3+Reranker，Token 降 60%）、语义缓存（用向量相似度判断是否是重复问题，命中缓存直接返回，API 调用降 30%）、模型降级路由（简单意图用 Qwen-7B，复杂推理才用 Qwen-72B，大模型调用量降 50%）。

**Q62: 最终降了多少成本？AI 投入前后带来的收益有多少？** [P0] *(ML公司)*

需要准备具体数字。典型回答："通过模型分级 + 缓存 + 上下文压缩，月度 Token 成本从 X 万降到 Y 万，降幅约 60%。业务收益：客服首次响应时间从平均 5 分钟降到 30 秒，人工坐席负担减少 40%，用户满意度提升 15%。"面试官想看到你有效果意识，不只是"用了 AI"，而是"用了 AI 并且量化了效果"。

**Q63: 个人一个月 token 使用量大概是多少？** [P2] *(HJ公司)*

这问的是你日常用 AI 工具（如 ChatGPT/Claude/Cursor）的消耗。诚实回答即可，比如"月均 200-500 万 Token，主要用于代码生成、文档撰写和技术调研"。

**Q64: 如果政府项目限制使用国外模型或工具，完全使用国内工具有没有影响？** [P1] *(HJ公司)*

回答："基本没有影响。国内模型（Qwen/DeepSeek/GLM）在中文理解和生成方面甚至优于国外模型。国内工具有完整替代方案：模型用 Qwen/DeepSeek、向量模型用 bge-m3、部署用 Ollama/vLLM、开发框架用 LangChain（开源无国界限制）。唯一影响是部分闭源工具（如 Claude Code）需要找国产替代。"

**Q65: 有没有做过国内外 AI 工具或模型的对比？** [P1] *(HJ公司)*

对比维度：中文能力（Qwen/DeepSeek 在中文场景通常优于 GPT/Claude）、代码能力（Claude 3.5 > GPT-4 > Qwen，但差距在缩小）、多模态（GPT-4V/Claude 领先，Qwen-VL 也不错）、成本（国内模型便宜 5-10 倍）、隐私（国内模型更适合政府/工业场景）、API 稳定性（国内模型在国内网络更稳定）。

**Q66: 同一个模型不同工具，或者同一个工具不同模型，效果差异有没有研究？** [P1] *(HJ公司)*

同一模型不同工具：差异主要在 Prompt 工程和工具集成。比如同样用 Qwen，Dify 的 Prompt 模板和 LangChain 的 Prompt 模板可能效果不同——因为系统提示、上下文组装方式、工具调用格式不同。同一工具不同模型：差异在模型能力本身。比如同样用 LangChain，GPT-4 的工具选择准确率高于 Qwen-7B，但成本也高 10 倍。

**Q67: 没用 AI 和用了 AI 后，你觉得效率提升有多大？** [P2] *(HJ公司)*

分场景回答：代码编写提升 40-60%（ boilerplate 和重复代码 AI 生成，复杂逻辑仍需人工）、文档撰写提升 60-80%（AI 生成初稿+人工修改）、技术调研提升 50%（AI 快速总结文档+人工验证）、学习新领域提升 70%（AI 做个性化导师）。整体来看 30-50% 的效率提升是合理的。

**Q68: 你会用什么 AI 工具来完成项目规划、草图、文档和开发？** [P2] *(HJ公司)*

项目规划：ChatGPT/Claude 做需求拆解和技术选型分析；草图：v0.dev 或 Cursor 做前端原型；文档：AI 生成初稿 + 人工校对；开发：Cursor/Claude Code 做 AI 辅助编程；测试：AI 生成单元测试用例。关键是展示你有完整的 AI 工具链。

**Q69: 任务重试会造成重复执行、重复调用大模型，你项目中如何做重复推理优化、降低成本？** [P1] *(HQ公司)*

重复推理优化：幂等设计（相同输入只执行一次，结果缓存）、请求去重（用 hash 指纹判断是否是重复请求）、断点续跑（任务中断后从最后成功的步骤恢复，不从头开始）、结果持久化（每个中间步骤的结果存库，重试时跳过已完成步骤）。Bull Queue 等消息队列支持 `attempts` 配置和死信队列。

## 六、SSE / WebSocket 实时通信

**Q78: SSE 和 WebSocket 分别说说？** [P0] *(FN公司/YF公司/HQ公司)*

SSE（Server-Sent Events）：基于 HTTP 的单向服务器推送。服务端通过 `Content-Type: text/event-stream` 持续推送数据，客户端用 `EventSource` 或 `fetch` 接收。优点：简单（基于 HTTP）、自动重连、支持断线续传（用 `Last-Event-ID`）、通过 HTTP 端口无需额外配置。缺点：单向（服务端→客户端）、不支持二进制。

WebSocket：全双工双向通信。通过 HTTP 升级握手建立持久连接。优点：双向、低延迟、支持二进制。缺点：更复杂（需要单独协议和端口配置）、不自动重连、连接管理更重。

选择：AI 流式输出用 SSE（只需要服务端→客户端推送，SSE 足够简单）；实时聊天/协同编辑用 WebSocket（需要双向通信）。

**Q79: SSE 断线续传？由于客户端因素断了，比如用户把窗口关掉了，又很快打开，或者刷新页面，让他继续接收 SSE 流，你们有没有做过？** [P1] *(YC公司)*

断线续传实现：服务端每个 SSE 事件带一个递增的 `event_id`，客户端保存最后收到的 `event_id`。重连时在请求头中带上 `Last-Event-ID: <last_id>`，服务端从该 ID 之后继续推送。如果运行已经结束（SSE 流关闭），需要后端支持"查询运行结果"接口，客户端重连后先拉取最终结果再恢复流。Nexbot 项目的设计可以参考——它有 `cursor-based replay + subscribe` 机制，断线后用 cursor 恢复。

**Q80: WebSocket 前端连接流程是什么？** [P1] *(SH公司)*

连接流程：创建 WebSocket 实例（`new WebSocket(url)`）→ 监听 `onopen`（连接成功）→ 监听 `onmessage`（接收消息）→ 监听 `onclose`（连接关闭）→ 监听 `onerror`（连接错误）→ 发送消息用 `ws.send(data)`。连接 URL 格式 `ws://` 或 `wss://`（加密）。

**Q81: WebSocket 怎么监听消息？怎么和后端约定事件？** [P1] *(SH公司)*

监听消息：`ws.onmessage = (event) => { const data = JSON.parse(event.data); ... }`。事件约定：通常用 JSON 格式约定消息类型，如 `{ "type": "chat_message", "payload": { "content": "hello" } }`。前端根据 `type` 字段分发到不同处理函数。后端也用相同格式推送。

**Q82: WebSocket 有没有做心跳/保活？断开后怎么重连？连接不上时怎么兜底？** [P1] *(SH公司)*

心跳：客户端定时（如 30 秒）发送 `{"type":"ping"}`，服务端回复 `{"type":"pong"}`，超时未收到 pong 则判定断开。重连：指数退避重连（1s → 2s → 4s → 8s → 最大 30s），避免短时间大量重连请求。兜底：重连失败超过 N 次后降级为 HTTP 轮询（每隔 5 秒拉取最新消息），并在 UI 上提示"连接已断开，正在使用离线模式"。

**Q83: 轮询频率怎么控制？有没有用指数退避？** [P1] *(SH公司)*

轮询频率控制：正常状态低频轮询（5-10 秒），有新消息时加快（1-2 秒），无新消息时减慢（10-30 秒自适应）。指数退避用于重连场景：初始 1 秒，每次失败翻倍，上限 30-60 秒，成功后重置。这样既保证用户体验又不会压垮服务端。

**Q84: 弱网情况下怎么提示用户？实时数据不准确时，前端怎么做交互提示？** [P1] *(SH公司)*

弱网提示：检测到延迟高或心跳超时时，UI 显示"网络状况不佳"的 Toast 或横幅。实时数据不准：显示数据的时间戳和"最后更新于 X 秒前"，让用户知道数据可能不是最新的；数据冲突时用乐观更新 + 回滚策略（先显示新数据，如果服务端确认失败则回滚）。

**Q85: 后端流式推送协议数据，前端如何实现边接收边流式渲染？** [P0] *(HQ公司)*

前端流式渲染流程：用 `fetch` 发起 POST 请求（不用 EventSource，因为需要 POST + 自定义头）→ 用 `ReadableStream` 读取 `response.body` → 用 `TextDecoder` 解码 → 按 SSE 格式（`data: ...\n\n`）分割帧 → 逐帧解析 JSON → 增量更新 UI。Nexbot 前端的 `sendStreamingMessageV2.ts` 就是这个实现——`fetch` SSE 解析 + `streamTurnAssembler.ts` 组装为 UI 块。

**Q86: AGUI 协议动态渲染表单/卡片整体实现流程？** [P1] *(HQ公司)*

AGUI（A2UI）是动态 UI 适配协议。流程：后端推送 `ui_event`（包含卡片类型和数据）→ 前端解析卡片类型（form/table/file/webpage）→ 根据类型路由到对应组件（UiFormCard/UiTableCard 等）→ 渲染卡片 → 用户交互（如提交表单）→ 通过 `client-events` 接口回传结果 → 后端恢复 Agent 执行。Nexbot 的 `card_render_tool` 就是这个机制的实现。

**Q87: 了解该协议底层 API 设计逻辑吗？** [P2] *(HQ公司)*

Nexbot 的卡片协议设计：`cardType` 字段区分类型（webpage/table/file/form），`data` 字段承载数据（url/columns+rows/fileRef/form_schema），`title`/`description` 是展示信息。前端有 `card-registry.ts` 做卡片类型注册，支持插件扩展自定义卡片。提交结果通过 `card_submit` metadata 传回后端，Agent 从 `runtime_context.request.metadata.card_submit` 读取。

## 七、前端基础

> 以下前端题给要点答案，详细展开请参考前端八股文资料。

**Q88: CSS3 你用得比较多的有哪些？渐变、旋转、缩放、动画这些你了解吗？** *(SH公司)*

常用：Flexbox 布局、Grid 布局、transition 过渡、transform 变换（rotate/scale/translate）、animation 动画（@keyframes）、border-radius 圆角、box-shadow 阴影、gradient 渐变（linear/radial）、media query 响应式。

**Q89: 做动画时，CSS 和 JS 分别怎么做？有什么区别？** *(SH公司)*

CSS 动画：用 @keyframes + animation 或 transition，声明式、浏览器优化、性能好（GPU 加速）。JS 动画：用 requestAnimationFrame 逐帧控制，命令式、更灵活、可以做复杂逻辑但性能不如 CSS。原则：简单动画用 CSS，需要复杂控制（如随交互动态变化）用 JS。

**Q90: Canvas 有没有用过？做过什么场景？** *(SH公司)*

Canvas 用于 2D 绘图。常见场景：数据可视化图表、图片编辑（裁剪/滤镜）、动画游戏、签名板。与 SVG 的区别：Canvas 是像素级绘制（适合高频更新），SVG 是矢量 DOM（适合交互丰富的静态图）。

**Q91: 红包/福袋动画是怎么做的？** *(SH公司)*

红包动画实现：CSS keyframes 做抖动/缩放效果，transform: rotate + scale 做打开效果。也可以用 Canvas/Lottie 做更精细的动画。打开红包的动作通常是用 CSS transform 做翻盖效果，不是画出来的。

**Q92: ES6 有哪些常用特性？** *(SH公司)*

let/const（块级作用域）、箭头函数、模板字符串、解构赋值、扩展运算符、Promise/async-await、Class 类、Module 模块（import/export）、Map/Set、默认参数、可选链?.和空值合并??。

**Q93: 什么是柯里化函数？** *(SH公司)*

柯里化（Currying）是把接受多个参数的函数转换为逐个接受参数的函数链。如 `f(a, b, c)` → `f(a)(b)(c)`。用途：参数复用、延迟执行、函数组合。实现：用闭包缓存已传参数，参数齐了才执行。

**Q94: ES5 和 ES6 构造函数分别怎么写？ES6 用什么构造对象？ES5 用什么？** *(SH公司)*

ES5：用构造函数 + prototype，`function Person(name) { this.name = name; } Person.prototype.say = function() {}`。ES6：用 class 语法，`class Person { constructor(name) { this.name = name; } say() {} }`。ES6 class 是 ES5 构造函数的语法糖，底层一样。

**Q95: Promise、扩展运算符、箭头函数、模板字符串这些是否了解？** *(SH公司)*

Promise：异步编程方案，三种状态 pending→fulfilled/rejected，then/catch/finally 链式调用。扩展运算符：`...` 展开数组/对象。箭头函数：`() => {}`，没有自己的 this/arguments。模板字符串：反引号 + ${} 插值。

**Q96: Vue 2 和 Vue 3 的区别是什么？** *(SH公司/YF公司)*

响应式：Vue2 用 Object.defineProperty（只能拦截属性，数组需特殊处理），Vue3 用 Proxy（全对象拦截，性能更好）。API：Vue2 Options API（data/methods/computed），Vue3 Composition API（setup + ref/reactive）。性能：Vue3 重写虚拟 DOM、静态提升、PatchFlag。Fragment：Vue3 支持多根节点。TypeScript：Vue3 原生支持。

**Q97: Object.defineProperty 和 Proxy 有什么区别？** *(SH公司)*

defineProperty：只能监听已有属性，新增/删除属性不触发（需要 $set/$delete）；数组索引修改不触发（需重写数组方法）。Proxy：监听整个对象，新增/删除属性都能触发；数组直接索引修改也能触发；性能更好（不需要递归遍历每个属性）。

**Q98: Vue 2 里数组/对象变更为什么可能不触发视图更新？如果数据变了但视图没变，可能是什么原因？** *(SH公司)*

原因：defineProperty 无法检测对象属性添加/删除和数组索引直接赋值。解决方案：用 `Vue.$set()` 添加新属性，用 `splice` 而非直接索引修改数组。数据变了视图没变的其他原因：异步更新（Vue 批量异步更新 DOM，用 $nextTick）、深拷贝问题、计算属性缓存。

**Q99: Vue2 响应式原理，数组方法是如何重写拦截的？哪些数组方法无法被原生拦截？** *(HQ公司)*

Vue2 重写了 push/pop/shift/unshift/splice/sort/reverse 七个变更方法，在方法执行后手动触发依赖通知。无法拦截的操作：直接索引赋值 `arr[0] = x`、直接修改长度 `arr.length = 0`。处理方式：用 $set 或 splice 替代。

**Q100: Vue3 中的 WeakMap/弱引用作用是什么？** *(HQ公司)*

Vue3 响应式系统用 WeakMap 存储对象→依赖映射（`targetMap = new WeakMap()`）。WeakMap 的键是弱引用——当对象没有其他引用时会被 GC 自动回收，不会造成内存泄漏。因为响应式系统可能关联大量临时对象，用 WeakMap 避免内存堆积。

**Q101: Vue 和 React 核心区别是什么？谁性能上限更高？** *(FN公司)*

核心区别：Vue 是响应式数据驱动（数据变→视图自动更新），React 是单向数据流 + 不可变状态（setState→重新渲染）。模板 vs JSX：Vue 用模板（编译时优化），React 用 JSX（运行时灵活）。API：Vue Composition API vs React Hooks。性能上限：React 灵活度更高但需要手动优化（memo/useMemo），Vue 自动优化但定制空间小。理论上 React 性能上限更高（可以精细控制渲染），但 Vue 开箱即用性能更好。

**Q102: React 和 Vue 最新版本在做什么？** *(FN公司)*

Vue 3.5：重构响应式系统（更细粒度的依赖追踪）、改进 SSR 性能。React 19：Actions（表单 action 属性）、use() Hook、Server Components 稳定、编译器（React Compiler 自动优化 memo）。

**Q103: 浏览器发起 HTTP 请求经历什么过程？哪些地方会发生缓存？** *(FN公司)*

请求过程：DNS 解析（域名→IP）→ TCP 三次握手 → TLS 握手（HTTPS）→ 发送 HTTP 请求 → 服务端处理 → 返回响应 → 浏览器渲染。缓存位置：DNS 缓存、浏览器缓存（强缓存 Cache-Control/Expires、协商缓存 ETag/Last-Modified）、CDN 缓存、Service Worker 缓存、代理缓存。

**Q104: 有做过浏览器兼容吗？低版本浏览器路由变了但页面不跳转怎么办？** *(SH公司)*

兼容处理：CSS 用 Autoprefixer 加前缀、用 Polyfill（core-js/babel-polyfill）补 JS API。低版本浏览器路由不变：可能因为 History API（pushState）不支持，降级为 Hash 路由（#/path）。框架（Vue/React Router）通常支持两种模式切换。

**Q105: 有没有做过上万条大数据量页面渲染？怎么优化？** *(YF公司)*

虚拟列表（Virtual Scroll）：只渲染视口内的元素（通常 20-50 个），滚动时动态替换。实现原理：总高度 = 数据量 × 行高，可视区域只渲染对应区间的 DOM。库：vue-virtual-scroller / react-window。其他优化：分页加载、Web Worker 后台处理、requestIdleCallback 分片渲染。

**Q106: 原生小程序如何做状态管理？setData 存在什么性能问题？** *(YF公司)*

小程序状态管理：可以用全局 App 全局数据、或引入 MobX/类 Vuex 库。setData 性能问题：每次 setData 都会触发 JS→Native→渲染层的通信和 DOM diff，大量数据或频繁调用会卡顿。优化：合并 setData 调用（不要循环里调）、只传变化的字段（不传整个对象）、复杂列表用 recycling-view。

**Q107: 首页加载慢、图片体积大，你会从哪些方向做优化？** *(YF公司)*

图片优化：WebP 格式（比 JPEG 小 30%）、懒加载（IntersectionObserver）、响应式图片（srcset 按设备加载不同尺寸）、CDN 图片压缩。加载优化：代码分割（路由懒加载）、预加载关键资源、Tree Shaking 减少包体积、Gzip/Brotli 压缩、SSR/SSG 首屏直出。

## 八、后端与数据库

**Q108: 怎么保证数据库和缓存之间的数据一致性？** *(YC公司)*

两种策略：Cache Aside（先更新数据库，再删除缓存，读时回写缓存）——最常用，接受短暂不一致。延迟双删（先删缓存→更新数据库→延迟再删缓存）——防止并发读写导致旧数据回写。最终一致性：用消息队列确保缓存最终一致。强一致性场景不用缓存或用分布式事务。大部分业务接受短暂不一致，Cache Aside 足够。

**Q109: 什么是数据库事务？结合你的项目，举例说明必须使用事务的业务场景？** *(HQ公司)*

事务（ACID）：原子性（全做或全不做）、一致性（数据约束不被破坏）、隔离性（并发互不干扰）、持久性（提交后不丢失）。必须用事务的场景：支付（扣款+生订单必须同时成功或失败）、库存扣减（减库存+创建订单）。在 AI 系统中：知识库入库（写文档+写向量+写索引，要么全成功要么全回滚）。

**Q110: MySQL 大表优化怎么做？百万级数据后出现慢插入。** *(TK公司)*

优化方向：索引优化（加合适索引、避免索引失效）、分表分库（水平分表按时间/ID 切分）、读写分离（主写从读）、分区表（PARTITION BY RANGE）、冷热分离（历史数据归档）。慢插入特别：检查是否有过多索引（插入时要更新所有索引）、考虑用批量插入替代逐条、检查磁盘 IO 是否瓶颈。

**Q111: MySQL5.6 和 8.0 有什么区别？** *(YF公司)*

主要区别：8.0 默认 utf8mb4（5.6 默认 latin1）、8.0 有窗口函数（OVER/PARTITION BY）、8.0 有 CTE 递归查询、8.0 JSON 类型增强、8.0 降序索引、8.0 不可见索引、8.0 角色（ROLE）权限管理、8.0 InnoDB 性能提升。

**Q112: 数据库索引的最左匹配原则是什么？** *(YF公司)*

联合索引 (a, b, c) 的 B+Tree 按 a→b→c 排序。查询条件必须从最左列开始才能命中索引：`WHERE a=1` 命中，`WHERE a=1 AND b=2` 命中，`WHERE b=2` 不命中（跳过了 a），`WHERE a=1 AND c=3` 部分命中（只用到 a）。

**Q113: 高并发下如何避免超卖？MySQL 怎么实现库存扣减？Redis 分布式锁或 Lua 脚本如何使用？** *(PY公司)*

防超卖方案：MySQL 乐观锁（`UPDATE stock SET count=count-1 WHERE id=1 AND count>0`，利用行锁保证不超卖）、Redis 分布式锁（SETNX + 过期时间，获取锁后操作，释放锁用 Lua 保证原子性）、Lua 脚本（在 Redis 中原子执行"检查库存+扣减"）。

**Q114: 限流、接口削峰、事务和乐观锁如何落地？** *(PY公司)*

限流：令牌桶（Guava RateLimiter）或滑动窗口（Redis + Lua）。削峰：消息队列（请求写入 Kafka/RabbitMQ，消费者按速率处理）。乐观锁：版本号机制（`UPDATE ... WHERE version=x`）。落地组合：网关限流→请求入队→消费者扣库存（乐观锁）→异步返回结果。

**Q115: MySQL 慢查询、索引、SQL 优化做过哪些？** *(HQ公司)*

慢查询排查：开启 slow_query_log、用 EXPLAIN 分析执行计划（看 type/key/rows/Extra）。索引优化：避免全表扫描（加索引）、避免索引失效（不在索引列做函数运算/类型转换/LIKE 左模糊）。SQL 优化：避免 SELECT *、大查询用 LIMIT、JOIN 用小表驱动大表、子查询改 JOIN。

**Q116: 有没有分库分表、读写分离、数据库深度调优的实战经验？** *(HQ公司)*

诚实回答。如果有经验讲具体方案；如果没有可以说"了解 ShardingSphere 等中间件的原理，实际项目中数据量还没到必须分库分表的程度，但做过读写分离（主从复制 + 读写分离中间件）和索引优化"。

**Q117: 使用 Bull Queue 处理长任务，进程崩溃/K8S 重启后任务会不会丢失？如何保证消息队列任务不丢失、保证任务一致性？** *(HQ公司)*

Bull Queue 基于 Redis。防丢失：开启任务持久化（Redis 配置 AOF/RDB）、任务状态写入数据库（不只是 Redis）、消费者处理完才 ACK（确认机制）。K8S 重启时：Pod 优雅终止（preStop hook 等待当前任务完成）、任务设置超时和重试、死信队列处理失败任务。断点续跑：任务步骤持久化到数据库，重启用 cursor 从最后成功的步骤恢复。

**Q118: 海豚工作流同步数据时 MySQL IO 爆红怎么排查？** *(TK公司)*

排查思路：先看 MySQL `SHOW PROCESSLIST` 看当前执行的慢 SQL；看 IO 瓶颈（`iostat -x 1` 看磁盘 IOPS/吞吐）；分析 SQL 执行计划（EXPLAIN）；三连表 JOIN 可能是笛卡尔积爆炸——检查 JOIN 条件和索引；插入报错可能是 undo log/redo log 写入瓶颈。优化：批量插入替代逐条、减少同步频率、加索引优化查询、分批处理。

**Q119: 微信支付接入流程？如何保证支付后到账？支付为什么用异步队列？回调验签流程？** *(PY公司)*

支付流程：下单（生成订单→调微信统一下单 API→获取预支付 ID）→ 前端拉起支付 → 用户支付 → 微信异步回调通知 → 验签 → 更新订单状态。保证到账：回调 + 主动查单（定时轮询未完成订单的支付状态）。异步队列原因：支付回调可能延迟、需要重试、不能阻塞微信回调请求。验签：用 API 密钥对回调参数做签名比对，防止伪造回调。队列挂掉：数据库兜底查单 + 对账。

**Q120: 怎么保证用户支付后积分/虚拟产品到账？如果队列挂掉，如何保证支付结果和系统状态一致？** *(PY公司)*

保证一致性：支付成功 → 写入"待入账"记录到数据库 → 投递消息到队列 → 消费者执行入账 → 更新记录为"已入账"。队列挂了：定时任务扫描"待入账超时"记录 → 主动查微信支付状态 → 已支付则手动入账。关键是有数据库兜底，不纯依赖队列。

**Q121: 假设设计一个秒杀系统，你会怎么设计架构？** *(PY公司)*

秒杀系统设计：前端静态化（CDN 缓存活动页）→ 限流（网关层令牌桶）→ 排队（请求写入消息队列削峰）→ 预扣库存（Redis Lua 原子扣减，不碰数据库）→ 异步下单（消费者从队列取请求，创建订单，扣数据库库存）→ 超时未支付回滚库存。关键点：库存放 Redis（快）、数据库做最终一致、队列削峰扛流量。

## 九、系统架构与高并发

**Q122: 从 0 到 1 搭建一个项目，怎么设计架构？** *(YC公司)*

设计步骤：需求分析（功能/非功能需求）→ 技术选型（语言/框架/数据库/中间件）→ 架构分层（入口→服务→存储）→ 模块拆分（按业务域拆）→ 接口设计（RESTful + 数据模型）→ 数据流设计（数据从哪来→怎么处理→存哪→怎么用）→ 部署方案（Docker/K8s）→ 非功能设计（限流/缓存/监控/日志）。关键原则：先跑通再优化（MVP 优先），不要过早设计。

**Q123: 传统 XX 业务系统的后端分几层？你搭框架的整体思路是什么？** *(HJ公司)*

经典分层：表现层（Controller/路由）→ 业务层（Service/Domain）→ 数据访问层（Repository/DAO）→ 基础设施层（数据库/缓存/MQ）。搭框架思路：先定目录结构（按业务域或按技术层）→ 定基础规范（编码规范/接口规范/错误处理）→ 搭基础设施（配置/日志/数据库连接/认证）→ 写第一个完整业务流（CRUD）作为模板 → 团队按模板复制扩展。

**Q124: 你们产品开发完后，最大用户并发数大概是多少？有没有做熔断之类的处理？** *(HJ公司)*

高并发处理：限流（网关层令牌桶/滑动窗口）、熔断（Sentinel/Hystrix，错误率超阈值自动熔断返回降级）、缓存（Redis 热点数据缓存）、异步队列（Kafka/RabbitMQ 削峰）、数据库读写分离、CDN 静态资源加速。Nexbot 的 Gateway 控制面就是完整范例——限流 20次/10s + 并发 ≤4 + 幂等 TTL 300s + 熔断 503 + 超时 90s。

**Q125: 全栈项目没有测试岗，是怎么测试的？有没有做过基于 AI 的界面测试？** *(HJ公司)*

无测试岗的测试策略：开发自测（单元测试 pytest/Jest）、集成测试（API 测试用 TestClient/Supertest）、E2E 测试（Playwright/Cypress）、手动冒烟测试。AI 界面测试：用 Playwright + AI 做 UI 自动化——AI 生成测试用例（描述页面交互场景→生成代码），AI 做断言（截图给 LLM 判断是否符合预期）。目前还不够成熟，可以辅助但不能完全替代人工。

**Q126: 如果客户是政府单位，要重新定制开发一个软件，包含客户端/APP/小程序，售前已完成调研，项目体量约 50 万，一个月内交付，你如何拆解流程？** *(HJ公司)*

拆解思路：需求确认（2 天，和售前对齐已有调研，确认范围和验收标准）→ 技术方案（1 天，选型+架构+接口定义）→ 并行开发（3 周：后端 API + 前端页面 + 小程序 + 客户端，分人并行）→ 联调测试（1 周）→ 部署上线（2 天）。关键：用 AI 辅助编码提效、用成熟框架（如 Vue+FastAPI）减少搭建时间、非核心功能降级为二期。

**Q127: 如果是资产管理系统，你会如何利用 AI 完成开发和交付？** *(HJ公司)*

AI 辅助开发和交付：需求阶段用 AI 拆解需求和生成 PRD；设计阶段用 AI 生成数据库表设计和 API 接口设计；编码阶段用 Cursor/Claude Code 生成 CRUD 代码（资产管理大量 CRUD 适合 AI 生成）；测试阶段用 AI 生成测试用例；文档阶段用 AI 生成用户手册。还可以在系统内嵌 AI 能力：资产智能搜索（RAG）、资产异常检测（ML）、自动生成资产报告（LLM）。

**Q128: 如何给几十上百个接口统一增加操作日志/接口监控？** *(HQ公司)*

统一处理用中间件/拦截器，不要每个接口手写。FastAPI 用 `@app.middleware("http")` 或依赖注入拦截所有请求，记录请求方法/路径/参数/耗时/状态码。NestJS 用 Interceptors 拦截。日志写入文件 + 异步推送到监控平台（ELK/Prometheus）。关键是一处拦截全局生效。

**Q129: NestJS 的依赖注入中，Provider 默认是单例还是每次请求新建？管道（Pipe）的概念和作用？** *(HQ公司)*

NestJS Provider 默认是单例（Scope = DEFAULT），整个应用共享一个实例。Request-scoped Provider 每次请求新建（但性能有损耗）。Pipe 管道用于数据转换和验证——在路由处理前对参数做校验（如 class-validator）或转换（如 ParseIntPipe）。类比 FastAPI 的 Pydantic 模型校验。

## 十、Docker / K8s 部署

**Q130: docker 和 k8s 的关系？容器和镜像的关系？怎么进行挂载的？** *(KJ公司)*

Docker 是容器运行时，K8s 是容器编排平台——K8s 管理 Docker 容器的调度、扩缩容、滚动更新。镜像是只读模板（含代码+环境），容器是镜像的运行实例（可读写层叠加在镜像上）。挂载：`docker run -v /host/path:/container/path` 把宿主机目录映射到容器内，数据持久化在宿主机。

**Q131: k8s 中多个服务怎么进行通讯？** *(KJ公司)*

K8s 服务通信：同集群内通过 Service（ClusterIP）访问，DNS 解析服务名到 ClusterIP；跨命名空间用 `服务名.命名空间.svc.cluster.local`。外部访问用 Ingress 或 NodePort。服务间调用直接用 HTTP 指向 Service 名称。也可以用 Service Mesh（Istio）做更精细的流量管理。

**Q132: 多租户的场景下 k8s 怎么部署的？一个租户一个对应 pod 吗？** *(JR公司)*

多租户部署方案：小规模用"命名空间隔离"（每个租户一个 Namespace，资源配额隔离）；中规模用"共享 Pod + 数据隔离"（所有租户共享 Pod，靠应用层 tenant_id 隔离数据）；大规模/高安全用"租户独立 Pod"（每个租户独立 Pod 甚至独立节点）。不是必须一个租户一个 Pod——取决于隔离级别要求。Nexbot 用的是应用层 tenant_id 隔离（所有租户共享服务，存储按 tenant_id 分目录）。

**Q133: K8S 日常用到什么程度？进程崩溃/K8S 重启后任务如何处理？** *(YF公司/HQ公司)*

K8s 核心概念：Pod（最小调度单元）、Deployment（管理 Pod 副本）、Service（服务发现）、Ingress（入口路由）、ConfigMap/Secret（配置管理）、PVC（持久化存储）。崩溃恢复：K8s 自动重启 Pod（restartPolicy: Always），但内存中的任务状态会丢失。解决：任务状态持久化到数据库/Redis，Pod 重启后从检查点恢复。
